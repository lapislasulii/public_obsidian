# Конспект лекции 23: Расширение структур данных



---

## 1. Обзор лекции

Лекция посвящена технике **расширения структур данных** — методологии, позволяющей добавлять дополнительные поля к существующим структурам с целью эффективного решения новых задач без потери асимптотических гарантий. В качестве базовой структуры используется красно-чёрное дерево: лектор сначала кратко повторяет его свойства и доказывает оценку высоты $O(\log n)$, после чего демонстрирует два классических расширения. Первое расширение — дерево порядковых статистик — позволяет находить $k$-й наименьший элемент и ранг произвольного узла за $O(\log n)$, храня в каждом узле размер его поддерева. Второе расширение — дерево отрезков (интервальное дерево) — решает задачу эффективного поиска пересекающихся интервалов и применяется, например, в планировщиках задач и диаграммах Ганта. Параллельно лектор вводит структуру B-деревьев, объясняет их мотивацию (дисковые операции, базы данных) и обсуждает практику командной разработки через GitHub.

---

## 2. Предварительные сведения

- **Бинарное дерево поиска (BST):** каждый узел содержит один ключ; левое поддерево меньше, правое больше.
- **Красно-чёрное дерево (RBT):** самобалансирующееся BST с пятью цветовыми правилами; высота $\Theta(\log n)$.
- **Инвариант BST:** `left->key <= key <= right->key`.
- **Балансировка:** повороты (rotate\_left / rotate\_right) и перекраска, сохраняющие BST-инвариант.
- **Порядковые статистики:** медиана, квантили, минимум, максимум по упорядоченному множеству.
- **Интервал:** пара $(low, high)$, $low < high$; два интервала пересекаются, если $a.low < b.high$ и $b.low < a.high$.
- **Страница диска (блок):** единица ввода-вывода на HDD/SSD; стоимость дискового доступа на порядки выше RAM.

## 3. Основное содержание

### 3.1. Красно-чёрное дерево: повторение и доказательство высоты

#### Определение

Красно-чёрное дерево — бинарное дерево поиска, в котором каждый узел окрашен в красный или чёрный цвет и выполняются пять свойств:

1. Каждый узел — красный или чёрный.
2. Корень — чёрный.
3. Все листья ($NIL$) — чёрные.
4. Оба ребёнка красного узла — чёрные (нет двух красных подряд).
5. Все простые пути от узла до листьев содержат одинаковое количество чёрных узлов.

#### Оценка высоты: $h \le 2\lfloor\log_2 n\rfloor$

**Интуиция (неформальное доказательство из лекции):**

1. Если в дереве присутствуют **только чёрные** узлы, свойство 5 требует, чтобы число чёрных узлов на всех путях от корня до листьев было одинаковым. Единственное такое дерево — **полное** бинарное дерево, высота которого $h_{black} = \lfloor\log_2 n\rfloor$.

2. Добавляем красные узлы. Свойство 4 запрещает два красных подряд, поэтому между любыми двумя красными узлами обязан стоять чёрный. Самая длинная возможная ветвь чередует цвета: чёрный–красный–чёрный–красный–… Это удваивает длину пути.

3. Следовательно, максимальная высота дерева:

$$h \le 2\lfloor\log_2 n\rfloor$$

Все операции (поиск, вставка, удаление) имеют сложность $O(\log n)$, поскольку они пропорциональны высоте.

| Дерево | Высота в худшем случае | Высота в лучшем случае |
|--------|------------------------|------------------------|
| BST (несбалансированное) | $O(n)$ | $O(\log n)$ |
| Красно-чёрное | $O(\log n)$ (точнее $2\log n$) | $O(\log n)$ |
| AVL | $O(\log n)$ | $O(\log n)$ |

### 3.2. Методология расширения структур данных

Лекция вводит **четырёхшаговую схему** расширения:

1. **Выбор базовой структуры** — чаще всего красно-чёрное дерево.
2. **Определение дополнительной информации** — какое новое поле нужно добавить в каждый узел.
3. **Проверка поддерживаемости** — можно ли обновлять новое поле за $O(\log n)$ при всех модифицирующих операциях (вставка, удаление, ротации).
4. **Разработка новых операций** — новые запросы, использующие дополнительное поле.

[дополнено]
Ключевое ограничение: дополнительное поле должно обновляться **локально** — то есть при вставке/удалении/ротации достаточно пересчитать значения в $O(1)$ узлах. Если обновление требует просмотра всего дерева, оно разрушает асимптотику. Классический критерий: поле `x.extra` должно определяться только через `x.left.extra`, `x.right.extra` и `x.key`.
[/дополнено]

### 3.3. Дерево порядковых статистик (Order-Statistic Tree)

#### Мотивирующая задача

Есть динамическое множество чисел (например, рост людей: $\{160, 165, 166, 168, 170, \ldots\}$), которое каждую секунду изменяется (добавление/удаление). Требуется при каждом изменении пересчитывать:

- медиану (квантиль 50%),
- квантиль 2.5%, квантиль 97.5%,
- квантили 25% и 75%,
- минимум, максимум.

Наивный подход: сортировать весь массив при каждом обновлении — $O(n \log n)$ на запрос. С деревом порядковых статистик каждое обновление и каждый запрос — $O(\log n)$.

[дополнено]
Практический пример: система мониторинга ошибок **Sentry** вычисляет квантили времени ответа (p50, p95, p99) в реальном времени над потоком событий. Дерево порядковых статистик — одна из структур, пригодных для этой задачи.
[/дополнено]

#### Структура узла

К стандартному красно-чёрному узлу добавляется поле `size` — размер поддерева, включая сам узел.

```c
struct OrderStatNode {
    int key;
    char color;
    int size;           // размер поддерева: 1 + left->size + right->size
    OrderStatNode* parent;
    OrderStatNode* left;
    OrderStatNode* right;
};
```

**Инвариант поля `size`:**

$$\texttt{node.size} = 1 + \texttt{node.left.size} + \texttt{node.right.size}$$

Для $NIL$-листьев: `size = 0`.

#### Поддержание инварианта при ротациях

При повороте затрагиваются ровно два узла, поэтому пересчёт за $O(1)$:

```c
// После rotate_left(x): x стал левым ребёнком y
y->size = x->size;
x->size = 1 + size(x->left) + size(x->right);
```

[дополнено]
При вставке поле `size` инкрементируется у всех предков нового узла (путь от вставленного узла до корня), что занимает $O(\log n)$ операций. При удалении — декрементируется аналогично.
[/дополнено]

#### Операция `get_rank`: ранг узла

**Ранг** узла — его индекс в отсортированном порядке (0-based или 1-based).

```c
int get_rank(OrderStatNode* node) {
    int rank = 0;
    if (node->left != nullptr)
        rank = node->left->size;   // все узлы левого поддерева меньше

    OrderStatNode* next = node;
    while (next != root) {
        if (next == next->parent->right) {
            // текущий узел — правый ребёнок: добавить левое поддерево родителя + сам родитель
            if (next->parent->left != nullptr)
                rank += next->parent->left->size;
            rank += 1;
        }
        next = next->parent;
    }
    return rank;
}
```

**Пример из слайда** (дерево с узлами 1, 6, 8.2, 8, 11, 17, 22, 25, 27):
- `node1` → ранг 0
- `node6` → ранг 1
- `node8.2` → ранг 2
- `node8` → ранг 3
- `node11` → ранг 4
- `node17` → ранг 7

**Сложность:** $O(\log n)$ — путь от узла до корня.

#### Операция `select`: поиск $k$-го элемента

```c
OrderStatNode* select(OrderStatNode* node, int index) {
    if (node == nullptr) return nullptr;

    int rank = 0;
    if (node->left != nullptr)
        rank = node->left->size;

    if (rank == index)
        return node;                          // нашли
    else if (index < rank)
        return select(node->left, index);     // ищем в левом поддереве
    return select(node->right, index - rank - 1);  // ищем в правом (сдвиг индекса)
}
```

#### Операция `quantile`: вычисление квантиля

```c
OrderStatNode* quantile(float q) {
    int index = int(root->size * q);   // например, q=0.5 → медиана
    return select(this->root, index - 1);
}
```

**Примеры из слайда:**
- квантиль 0.05 → узел со значением 1
- квантиль 0.50 → узел со значением 13 (медиана)
- квантиль 0.95 → узел со значением 27

**Сложность операций:**

| Операция | Сложность |
|----------|-----------|
| `insert` | $O(\log n)$ |
| `delete` | $O(\log n)$ |
| `get_rank` | $O(\log n)$ |
| `select` | $O(\log n)$ |
| `quantile` | $O(\log n)$ |

[дополнено]
**Типичная ошибка:** не обновлять поле `size` после ротации. После каждого вращения необходимо пересчитать `size` у обоих затронутых узлов — сначала у того, что опустился, потом у того, что поднялся. Порядок важен, так как поднявшийся узел использует `size` опустившегося.
[/дополнено]

### 3.4. Дерево отрезков (интервальное дерево, Interval Tree)

#### Мотивирующая задача

**Диаграмма Ганта:** есть множество работ (интервалов) вида $[low, high]$:

```
[[1, 2], [3, 7], [5, 6], [8, 10], ...]
```

Требуется эффективная структура данных для:
- хранения и обновления интервалов,
- поиска всех интервалов, пересекающихся с заданным.

#### Определение пересечения интервалов

Два интервала $a = [a_{low}, a_{high}]$ и $b = [b_{low}, b_{high}]$ пересекаются тогда и только тогда, когда:

$$a_{low} < b_{high} \quad \text{и} \quad b_{low} < a_{high}$$

Интервалы **не** пересекаются, если один заканчивается раньше, чем начинается другой:

$$a_{high} \le b_{low} \quad \text{или} \quad b_{high} \le a_{low}$$

#### Расширение красно-чёрного дерева

Базовая структура: красно-чёрное дерево, ключ узла — **левый конец интервала** $low$ (по нему строится дерево поиска).

Дополнительное поле: `max` — максимальный **правый конец** $high$ среди всех интервалов в поддереве данного узла.

```c
struct Interval {
    int min, max;       // [min, max)
    std::string key;

    Interval(int min_, int max_) {
        if (min_ >= max_)
            throw std::invalid_argument("max must be more than min");
        min = min_;
        max = max_;
    }

    std::string str() const {
        std::ostringstream oss;
        oss << "[" << min << ", " << max << "]";
        return oss.str();
    }

    // Оператор пересечения
    bool operator&(Interval i_) {
        Interval i = *this;
        bool intersect = i_.min < i.min && i_.max > i.max
                       | i_.min > i.min && i_.max < i.max;
        intersect |= i_.min < i.min && i_.max > i.min
                   | i_.min < i.max && i_.max > i.max;
        return intersect;
    }
};

struct IntervalNode {
    char color;
    Interval interval;
    int max;            // максимальный right конец в поддереве
    IntervalNode* parent;
    IntervalNode* left;
    IntervalNode* right;

    IntervalNode(char color_, Interval interval_)
        : color(color_), interval(interval_) {
        max = interval_.max;
    }

    ~IntervalNode() {
        delete left;
        delete right;
    }
};
```

**Инвариант поля `max`:**

$$\texttt{node.max} = \max(\texttt{node.interval.high},\; \texttt{node.left.max},\; \texttt{node.right.max})$$

#### Алгоритм поиска пересекающегося интервала

```c
IntervalNode* interval_search(Interval interval) {
    IntervalNode* x = this->root;

    while (x != nullptr && !(interval & x->interval)) {
        bool go_left = x->left != nullptr;
        go_left &= x->left->max >= interval.min;
        x = go_left ? x->left : x->right;
    }

    return x;   // nullptr — пересечения нет; иначе — найденный узел
}
```

**Интуиция алгоритма:**

- Если левое поддерево существует и его `max >= interval.low`, то в нём **гарантированно** может быть пересекающийся интервал (иначе там нечего искать).
- Идём влево, если условие выполнено, иначе — вправо.
- Инвариант: если пересечение существует где-то в дереве, то алгоритм его найдёт.

**Сложность:** $O(\log n)$ — один проход от корня до листа.

[дополнено]
**Почему идти влево корректно?** Если `x->left->max < interval.low`, то в левом поддереве нет ни одного интервала, правый конец которого достигает `interval.low`. Поскольку пересечение требует `node.high > interval.low`, левое поддерево целиком не пересекается с запросом. Такое рассуждение — прямое следствие хранения поля `max`.
[/дополнено]

**Пример из слайда:** запрос $[22, 24]$ — алгоритм проходит от корня, проверяет поле `max` в левом поддереве, принимает решение о направлении, и находит пересекающийся интервал.

[дополнено]
**Типичная ошибка:** при вставке или удалении узла забыть обновить поле `max` у всех предков вверх по дереву. После ротации также нужно пересчитать `max` у двух затронутых узлов (сначала опустившегося, потом поднявшегося).
[/дополнено]

### 3.5. B-деревья: обзор и мотивация

#### Почему не красно-чёрное дерево для баз данных?

Красно-чёрное дерево хранится в оперативной памяти. Базы данных работают с диском (HDD/SSD). Каждый доступ к диску — это блочная операция, несравнимо медленнее RAM. При $n = 10^6$ элементов RBT имеет высоту $\approx 40$, то есть до 40 дисковых операций на поиск.

**Redis** хранит данные в RAM — там RBT подходит.  
**PostgreSQL / SQLite** (и подобные) хранят данные на диске — там нужно B-дерево.

#### Идея B-дерева

B-дерево — сбалансированное дерево, в котором **каждый узел содержит несколько ключей и несколько детей**. Это сокращает высоту дерева и, следовательно, число дисковых операций при поиске.

**Ключевой параметр:** $t$ — минимальная степень (задаётся при инициализации дерева). Чем больше $t$, тем "шире" и ниже дерево.

| Характеристика | Значение |
|----------------|----------|
| Мин. ключей в узле (кроме корня) | $t - 1$ |
| Макс. ключей в узле | $2t - 1$ |
| Мин. детей (кроме корня) | $t$ |
| Макс. детей | $2t$ |
| Высота дерева | $O(\log_t n)$ |

**Почему ключей на 1 меньше, чем детей?** Ключи делят числовую прямую на отрезки: $(-\infty, k_1)$, $(k_1, k_2)$, ..., $(k_{m-1}, +\infty)$. При $m$ ключах получается $m+1$ "щель" — ровно столько детей.

**Пример для $t = 3$:**
- Максимум ключей: $2 \cdot 3 - 1 = 5$
- Максимум детей: $2 \cdot 3 = 6$

#### Операция split (разделение узла)

Когда узел заполнен (содержит $2t - 1$ ключей), при вставке нового элемента:
1. Средний ключ поднимается к родителю.
2. Узел делится на два дочерних.

Это единственная "тяжёлая" операция; в среднем она редка, поэтому вставка в B-дерево амортизированно эффективна.

#### Сравнение RBT, AVL и B-дерева

| Структура | Ключей в узле | Детей | Высота | Применение |
|-----------|---------------|-------|--------|------------|
| BST | 1 | 2 | $O(n)$ — $O(\log n)$ | Обучение |
| AVL-дерево | 1 | 2 | $O(\log n)$ | In-memory, строгий баланс |
| Красно-чёрное | 1 | 2 | $\le 2\log_2 n$ | In-memory индексы (std::map) |
| B-дерево (степень $t$) | $t{-}1 \ldots 2t{-}1$ | $t \ldots 2t$ | $O(\log_t n)$ | Дисковые СУБД, файловые системы |

[дополнено]
**Физическая оптимизация.** Размер узла B-дерева выбирается кратным размеру блока диска (обычно 4–16 КБ), чтобы за одну дисковую операцию считывался ровно один узел. Инженеры баз данных исторически подгоняли $t$ под скорость вращения диска и размер сектора. Та же логика применима к распределённым системам: сетевой трафик между дата-центрами — аналог дискового доступа, и B-деревья минимизируют число обращений.
[/дополнено]

## 5. Справочный лист

### Красно-чёрное дерево

- 5 свойств: (1) цвет, (2) корень чёрный, (3) листья ($NIL$) чёрные, (4) нет двух красных подряд, (5) одинаковое число чёрных на всех путях до листьев.
- Высота: $h \le 2\lfloor\log_2 n\rfloor$.
- Все операции: $O(\log n)$.

### Дерево порядковых статистик

- Дополнительное поле: `size = 1 + left.size + right.size` (для $NIL$: `size = 0`).
- `select(node, i)` — найти узел с индексом $i$: $O(\log n)$.
- `get_rank(node)` — ранг узла: $O(\log n)$.
- `quantile(q)` — квантиль $q \in [0, 1]$: вычислить `index = root.size * q`, затем `select(root, index)`: $O(\log n)$.
- Обновление `size` при ротации: $O(1)$ (два узла).

### Интервальное дерево

- Ключ узла: `interval.low` (нижняя граница).
- Дополнительное поле: `max = max(interval.high, left.max, right.max)`.
- Условие пересечения: $a.low < b.high$ И $b.low < a.high$.
- `interval_search(i)`: идти влево, если `left.max >= i.low`; иначе вправо: $O(\log n)$.
- Обновление `max` при ротации: $O(1)$ (два узла).

### B-дерево

- Параметр $t \ge 2$ задаётся при создании.
- Кол-во ключей в узле: $[t-1,\; 2t-1]$ (корень: $[1,\; 2t-1]$).
- Кол-во детей: $[t,\; 2t]$ (для внутренних узлов).
- Высота: $O(\log_t n)$.
- `split(node)`: средний ключ уходит к родителю, узел делится пополам.
- Все листья на одном уровне (дерево строго сбалансировано по высоте).
- Размер узла подбирается под размер страницы диска.

## 6. Практические советы

1. **Сначала — структура и сигнатуры, потом — реализация.** При командной разработке прежде всего синхронизируйте `struct` и заголовки функций (`typedef`, сигнатуры). Только после этого разбивайтесь на подзадачи — это исключает конфликты при слиянии кода.

2. **Атомарность каждой функции.** Каждый метод (вставка, удаление, поиск) должен быть реализован и протестирован независимо от остальных. Функция удаления не должна зависеть от незаконченной функции вставки.

3. **Выносите повторяющиеся условия в helper-функции.** Вместо проверки `node->keys_count >= 2 * t - 1` в пяти местах напишите `bool is_full(BNode* node, int t)` — это уменьшает число ошибок при изменении формулы.

4. **Обновляйте дополнительные поля при каждой ротации.** Самая частая ошибка при расширении RBT — забытый пересчёт `size` или `max` после rotate\_left/rotate\_right. Всегда сначала обновляйте опустившийся узел, затем поднявшийся.

5. **Используйте визуализаторы для отладки.** Перед написанием кода поиграйте с анимированными реализациями в браузере (JS-деревья): это помогает закрепить алгоритм вставки/удаления и заметить пограничные случаи до того, как вы столкнётесь с ними в коде.
