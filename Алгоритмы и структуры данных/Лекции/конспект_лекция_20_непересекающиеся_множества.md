# Конспект лекции 20: Структуры данных для непересекающихся множеств (Disjoint Set Union)



---

## 1. Обзор лекции

Лекция посвящена структуре данных «непересекающиеся множества» (Disjoint Set Union, DSU), которая позволяет эффективно поддерживать разбиение элементов на непересекающиеся группы. Рассматривается мотивация через задачу кластеризации: каждый объект изначально является собственным кластером, и в процессе работы алгоритма кластеры сливаются. Последовательно разбираются четыре реализации с нарастающей сложностью: через словарь, через связный список, через дерево и через дерево со сжатием путей. Параллельно с основной темой лекция затрагивает амортизационный анализ динамического массива — метод потенциала и метод банковских монет — как инструмент для получения средней оценки стоимости операции. В итоге достигается почти константная амортизированная сложность $O(\log^* n)$ для обеих ключевых операций DSU.

---

## 2. Предварительные сведения

- **Множество и разбиение:** коллекция элементов, каждый из которых принадлежит ровно одному подмножеству.
- **Словарь (хеш-таблица):** доступ по ключу за $O(1)$ в среднем.
- **Связный список:** последовательная структура, обход за $O(n)$.
- **Дерево (лес):** иерархическая структура; высота сбалансированного дерева из $n$ узлов равна $O(\log n)$.
- **Амортизационный анализ:** способ оценить среднюю стоимость одной операции в последовательности из $n$ операций, а не стоимость худшего случая каждой из них.
- **Метод банковских монет (Banker's method):** каждой операции назначается «амортизированная стоимость»; разница между ней и реальной стоимостью откладывается в «банк» для оплаты будущих дорогих операций.
- **Метод потенциала:** вводится функция $\Phi$ (потенциал) состояния структуры; амортизированная стоимость $i$-й операции: $\hat{c}_i = c_i + \Phi_i - \Phi_{i-1}$.
- **Геометрическая прогрессия:** сумма $1 + 2 + 4 + \ldots + 2^k < 2 \cdot 2^k$.
- **Итерированный логарифм $\log^* n$:** количество раз, которое нужно применить логарифм к $n$, пока результат не станет меньше 1. Растёт крайне медленно: для $n = 2^{65536}$ значение $\log^* n = 5$.

---

## 3. Основное содержание

### 3.1. Мотивация: задача кластеризации

**Определение.** Задача кластеризации — разбить набор объектов на группы (кластеры) по некоторому критерию близости. На каждом шаге алгоритм проверяет, принадлежат ли два объекта одному кластеру, и при необходимости сливает два кластера в один.

**Свойства DSU:**
- Каждый элемент принадлежит ровно одному множеству.
- Множества попарно не пересекаются.

**Операции DSU:**
| Операция | Описание |
|----------|----------|
| `ADD(x)` | Добавить элемент `x` как новое одноэлементное подмножество |
| `FIND(x)` | Найти представителя (head) подмножества, которому принадлежит `x` |
| `UNION(a, b)` | Объединить подмножества, содержащие `a` и `b` |

---

### 3.2. Реализация через словарь

**Механизм.** Словарь `DSU` хранит пары `{элемент -> представитель множества}`. Представитель (head) каждого множества — один из его элементов; все элементы одного множества указывают на этот же head.

**Пример.** Для множеств $\{\{1, 5\}, \{6\}, \{8, 9\}\}$:

| Индекс | Ссылка (head) |
|--------|--------------|
| 1      | 1            |
| 5      | 1            |
| 6      | 6            |
| 8      | 8            |
| 9      | 8            |

**Код:**

```python
DSU: dict[int, int] = {}

def find(num: int) -> int | None:
    # Возвращаем head подмножества
    if num not in DSU:
        return None
    return DSU[num]

def union(a: int, b: int):
    a, b = find(a), find(b)
    for k, v in DSU.items():
        if v == b:
            DSU[k] = a
```

**Сложность:**
- `FIND`: $O(1)$ — прямой доступ по ключу.
- `UNION`: $O(n)$ — полный проход по всему словарю для перезаписи ссылок.

**Интуиция.** Поиск мгновенный, но объединение требует перебора всех элементов — это узкое место.

**Типичные ошибки:**
- Не обновить ссылки всех элементов при `UNION`: часть элементов останется привязана к старому head.

---

### 3.3. Реализация через словарь + связный список

**Механизм.** Значение в словаре теперь — не целое число, а ссылка на связный список, содержащий все элементы множества. При `UNION` итерируемся по меньшему из двух списков и присоединяем его элементы к большему, обновляя словарь.

**Код:**

```python
def find(num: int) -> LinkedList | None:
    if num not in DSU:
        return None
    return DSU[num]  # LinkedList

def union(a: int, b: int):
    a, b = find(a), find(b)  # LinkedList
    for value in b:
        a.add(value)
        DSU[value] = a
```

**Сложность (без оптимизации):**
- `FIND`: $O(1)$
- `UNION`: $O(n)$ — в худшем случае весь второй список

**Вывод.** Относительно реализации на словаре без связного списка выигрыша нет.

---

### 3.4. Улучшение: объединение по размеру (Weighted Union / Union by Size)

**Механизм.** При `UNION` всегда итерируемся по **меньшему** списку, присоединяя его к большему. Это гарантирует, что каждый элемент может быть перемещён не более $\log n$ раз.

**Код:**

```python
def union(a: int, b: int):
    a, b = find(a), find(b)
    # a остаётся большим, b — меньшим
    a, b = (a, b) if len(b) < len(a) else (b, a)
    for value in b:  # итерируем меньший список
        a.add(value)
        DSU[value] = a
```

**Анализ сложности.**

Рассмотрим серию из $n - 1$ операций `UNION`, пока все элементы не окажутся в одном множестве.

- **Лучший случай (линейная цепочка):** каждый раз объединяем множества размером 1 и $k$. Суммарная работа: $1 + 1 + \ldots = O(n)$.
- **Худший случай:** каждый раз объединяем два равных множества. Элемент может быть перемещён только тогда, когда его множество становится частью множества хотя бы вдвое большего. Значит, каждый элемент перемещается не более $\log_2 n$ раз. Суммарная работа: $O(n \log n)$, т.е. амортизированно $O^*(\log n)$ на операцию.

**Итоговые сложности:**
- `FIND`: $O(1)$
- `UNION`: $O^*(\log n)$ амортизированно

---

### 3.5. Реализация через лес деревьев

**Механизм.** Каждое подмножество — дерево, корень которого является представителем. Каждый узел хранит ссылку на родителя; у корня `parent == self`. Словарь `DSU_tree` отображает значение на узел дерева.

```python
class Node:
    def __init__(self, val):
        self.val_ = val
        self.parent_ = self  # изначально узел — корень
        self.size_ = 1

DSU_tree: dict[int, Node] = {}
```

**`find_tree` — поиск корня:**

```python
def find_tree(num: int) -> Node | None:
    if num not in DSU_tree:
        return None
    node = DSU_tree[num]
    while node.val_ != node.parent_.val_:
        node = node.parent_
    return node
```

**`union_tree` — объединение по размеру:**

```python
def union_tree(a: int, b: int):
    a, b = find_tree(a), find_tree(b)  # Node (корни)
    # a — большее дерево, b — меньшее
    a, b = (a, b) if b.size_ <= a.size_ else (b, a)
    b.parent_ = a          # корень меньшего → дочерний к большему
    a.size_ = a.size_ + b.size_
```

**Сложность.**

При использовании **объединения по размеру** высота любого дерева не превышает $\log_2 n$. Доказательство: при каждом объединении поддерево, которое «опускается» вглубь, как минимум удваивается по размеру. Значит, глубина узла ограничена $\log_2 n$.

- `FIND`: $O(\log n)$ — подъём по дереву от листа до корня.
- `UNION`: $O(\log n)$ — включает вызов `FIND`.

**Интуиция.** Дерево играет роль компактного представления цепочки принадлежности: вместо перебора всего списка мы идём вверх по ссылкам на родителя.

**Типичные ошибки:**
- Не обновлять `size_` при объединении.
- Присоединять большее поддерево к меньшему — это приводит к разбалансировке и деградации до $O(n)$.

---

### 3.6. Оптимизация: сжатие путей (Path Compression)

**Механизм.** При каждом вызове `FIND` все узлы на пути от запрошенного элемента до корня **перевешиваются напрямую к корню**. Последующие обращения к тем же элементам станут за $O(1)$.

**Иллюстрация.**

До сжатия:
```
корень
  |
  узел A
    |
    узел B
      |
      узел C  ← find(C) поднимается через B, A до корня
```

После `find_tree(C)` со сжатием:
```
корень
 /|\
A B C   ← все узлы теперь напрямую к корню
```

**Модифицированный `find_tree` (рекурсивный вариант):**

```python
def find_tree(num: int) -> Node | None:
    if num not in DSU_tree:
        return None
    node = DSU_tree[num]
    if node.parent_ == node:
        return node
    node.parent_ = find_tree(node.parent_.val_)  # сжатие пути
    return node.parent_
```

**Сложность.**

Совместное применение **объединения по размеру** и **сжатия путей** даёт амортизированную сложность $O(\log^* n)$ на каждую операцию, где $\log^* n$ — итерированный логарифм.

[дополнено]
Точное доказательство использует функцию Аккермана и даёт оценку $O(\alpha(n))$ на операцию, где $\alpha(n)$ — обратная функция Аккермана. На практике для любых реальных $n$ выполняется $\alpha(n) \le 4$, поэтому структура считается практически константной по времени. $\log^* n$ является более слабой, но интуитивно более понятной оценкой.
[/дополнено]

**Значения $\log^* n$:**

| $n$ | $\log^* n$ |
|-----|-----------|
| 1 | 0 |
| 2 | 1 |
| 4 | 2 |
| 16 | 3 |
| 65536 | 4 |
| $2^{65536}$ | 5 |

Пример из лекции: $10^{80} \to \log(10^{80}) = 80 \to \log(80) \approx 1{,}9 \to \log(1{,}9) \approx 0{,}2 < 1$ — итого **3 шага**.

### 3.7. Амортизационный анализ динамического массива (вводный раздел лекции)

Эта часть лекции разбирается как инструментальный фон для понимания анализа DSU.

#### 3.7.1. Постановка задачи

Динамический массив (`ArrayList` / `vector`) при заполнении выделяет новый блок памяти (обычно вдвое большего размера) и копирует все элементы. Стоимость отдельной вставки варьируется от $O(1)$ до $O(n)$. Нас интересует **средняя** стоимость одной вставки на последовательности из $n$ операций.

#### 3.7.2. Метод суммирования (агрегатный метод)

При удвоении размера копирование происходит на шагах $1, 2, 4, 8, \ldots$ Суммарная работа на копирование:

$$1 + 2 + 4 + \ldots + 2^{\lfloor \log_2 n \rfloor} < 2n$$

Так как последний член геометрической прогрессии $< n$, а сумма всех предыдущих членов тоже $< n$ (свойство геометрической прогрессии), суммарная работа $< 2n$.

Итого: $n$ вставок $+ 2n$ копирований $= 3n$. Амортизированная стоимость одной вставки:

$$\hat{c} = \frac{3n}{n} = 3 = O(1)$$

#### 3.7.3. Метод банковских монет (Banker's method / метод монет)

**Идея.** Каждой операции назначается «амортизированная стоимость» — количество монет, которое она «зарабатывает». Часть монет тратится на реальную работу, остаток кладётся в «банк» для оплаты будущих дорогих операций.

**Правило для динамического массива (удвоение).**

- Каждой вставке назначаем **3 монеты**.
- 1 монета тратится на саму вставку.
- 2 монеты кладутся в банк.

При заполнении массива (capacity = $m$, size = $m$) в банке накоплено $2 \cdot \frac{m}{2} = m$ монет (потому что $m/2$ последних вставок принесли по 2 монеты). Этих монет хватает ровно на копирование $m$ элементов в новый массив.

**Проверка корректности:**
- Баланс банка никогда не уходит в минус — правило работает.
- Амортизированная стоимость = константа 3 = $O(1)$.

#### 3.7.4. Метод потенциала

**Формулировка.** Определяем функцию потенциала $\Phi(S)$ для состояния структуры $S$. Амортизированная стоимость $i$-й операции:

$$\hat{c}_i = c_i + \Phi_i - \Phi_{i-1}$$

где $c_i$ — реальная стоимость, $\Phi_i$ — потенциал после операции, $\Phi_{i-1}$ — потенциал до.

**Требования к функции потенциала:**
1. $\Phi_0 = 0$ (начальный потенциал равен нулю либо — более слабое требование — $\Phi_i \ge \Phi_0$ для всех $i$).
2. Сумма $\hat{c}_i$ должна давать корректную верхнюю оценку суммарной реальной работы.

**Потенциал для динамического массива (удвоение):**

$$\Phi = 2n - m$$

где $n$ = `size` (количество элементов), $m$ = `capacity` (выделенная ёмкость).

**Расчёт для обычной вставки** (без расширения, $n < m$):
- Реальная работа: $c_i = 1$.
- $\Phi_i - \Phi_{i-1} = (2(n+1) - m) - (2n - m) = 2$.
- Амортизированная стоимость: $\hat{c}_i = 1 + 2 = 3$.

**Расчёт для вставки с расширением** ($n = m$, новая ёмкость $m' = 2m$):
- Реальная работа: $c_i = n + 1 = m + 1$ (копируем $m$ элементов + вставляем 1).
- $\Phi_i - \Phi_{i-1} = (2(n+1) - 2m) - (2n - m) = 2n + 2 - 2m - 2n + m = 2 - m$.
- Амортизированная стоимость: $\hat{c}_i = (m+1) + (2 - m) = 3$.

Амортизированная стоимость в обоих случаях = **3 = $O(1)$**. Это соответствует результату агрегатного метода.

[дополнено]
Если массив расширяется на $+1$ (линейный рост) вместо удвоения, то суммарная работа копирования равна $1 + 2 + 3 + \ldots + n = \frac{n(n+1)}{2} = O(n^2)$, и амортизированная стоимость составит $O(n)$ — это плохо. Именно поэтому для динамических массивов используется геометрический рост (обычно $\times 2$).
[/дополнено]

## 4. Сравнительная таблица реализаций DSU

| Реализация | `FIND` | `UNION` | Примечание |
|-----------|--------|---------|-----------|
| Словарь (прямая ссылка) | $O(1)$ | $O(n)$ | Полный перебор словаря |
| Словарь + связный список | $O(1)$ | $O(n)$ | Без оптимизации |
| Словарь + список + union by size | $O(1)$ | $O^*(\log n)$ | Амортизированно |
| Дерево + union by size | $O(\log n)$ | $O(\log n)$ | Ограниченная высота |
| Дерево + union by size + path compression | $O(\log^* n)$ | $O(\log^* n)$ | Практически $O(1)$ |

$O^*(\log n)$ — амортизированная оценка на операцию (суммарная работа $O(n \log n)$ на $n$ операций).

## 5. Справочный лист

### Операции DSU

- `FIND(x)`: вернуть представителя (корня) множества, содержащего `x`.
- `UNION(a, b)`: объединить множества `FIND(a)` и `FIND(b)`.
- `ADD(x)`: создать одноэлементное множество $\{x\}$.

### Ключевые сложности

| Структура / метод | `FIND` | `UNION` |
|------------------|--------|---------|
| Словарь | $O(1)$ | $O(n)$ |
| Список + union by size | $O(1)$ | $O^*(\log n)$ |
| Дерево + union by size | $O(\log n)$ | $O(\log n)$ |
| Дерево + union by size + path compression | $O(\log^* n)$ | $O(\log^* n)$ |

### Потенциал динамического массива

$$\Phi = 2n - m, \quad n = \text{size}, \quad m = \text{capacity}$$

Амортизированная стоимость вставки (при удвоении): $\hat{c} = 3 = O(1)$.

### Итерированный логарифм

$$\log^* n = \min\{k \ge 0 : \underbrace{\log \log \ldots \log}_{k} n \le 1\}$$

Для всех практических $n$: $\log^* n \le 5$.

### Метод потенциала — формула

$$\hat{c}_i = c_i + \Phi_i - \Phi_{i-1}$$

Условие корректности: $\Phi_i \ge \Phi_0$ для всех $i \ge 0$.

### Свойство геометрической прогрессии

$$1 + 2 + 4 + \ldots + 2^k = 2^{k+1} - 1 < 2 \cdot 2^k$$

Если последний член $< n$, то и вся сумма $< 2n$.

---

## 6. Практические советы

1. **Всегда применяйте оба улучшения одновременно.** Union by size отдельно даёт $O(\log n)$; path compression отдельно даёт $O(\log n)$ амортизированно. Только вместе они дают $O(\log^* n) \approx O(1)$.

2. **Не меняйте местами большее и меньшее при `UNION`.** При union by size корень меньшего поддерева становится дочерним к корню большего. Ошибка инверсии немедленно разрушает гарантию высоты.

3. **При выборе функции потенциала проверяйте два условия:** (а) $\Phi_i \ge \Phi_0$ для всех $i$; (б) амортизированная стоимость константна для каждого типа операции. Если хотя бы одно условие нарушено, функция потенциала подобрана неверно.

4. **Для динамических структур предпочитайте геометрический рост ёмкости.** Удвоение даёт $O(1)$ амортизированную стоимость вставки; линейный рост на $+1$ — $O(n)$ на вставку.

5. **DSU применяйте в задачах о связности.** Типичные сценарии: алгоритм Краскала для минимального остовного дерева, задачи на offline-обработку запросов о связности в графе, кластеризация данных.
