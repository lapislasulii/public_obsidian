## 1. Обзор лекции

Лекция посвящена практической реализации и теоретическому анализу алгоритмов обхода и обработки графов на языке C/C++. Начинается с живого написания кода базовых операций с графом: создания графа, добавления узлов, добавления рёбер и поиска узла по идентификатору. Далее рассматриваются два ключевых алгоритма обхода — BFS (поиск в ширину) и DFS (поиск в глубину), включая итеративную версию DFS через явный стек. Лекция охватывает топологическую сортировку ориентированных ациклических графов (DAG) с конкретным примером кода и обсуждением практического применения в task-планировщиках. В завершение кратко вводятся алгоритмы нахождения кратчайших путей: Дейкстра, A\*, Беллман–Форд и Флойд–Уоршелл — с акцентом на различия в области применения.

## 2. Предварительные сведения

- **Граф** — множество вершин (узлов) и рёбер (связей) между ними.
- **Ориентированный граф (орграф)** — граф, в котором рёбра имеют направление.
- **Ациклический граф (DAG)** — ориентированный граф без циклов.
- **Связный граф** — граф, в котором между любыми двумя вершинами существует путь.
- **Список смежности** — способ хранения графа: для каждого узла хранится список его соседей.
- **Матрица смежности** — двумерный массив $V \times V$, где $dp[u][v]$ = вес ребра $u \to v$.
- **BFS (Breadth-First Search)** — обход графа уровнями, использует очередь (FIFO).
- **DFS (Depth-First Search)** — обход графа вглубь, использует стек (LIFO) или рекурсию.
- **Хеш-таблица / хеш-сет** — структура для хранения посещённых узлов с $O(1)$ поиском и добавлением.
- **Очередь с приоритетом (min-heap)** — структура, возвращающая элемент с наименьшим ключом, используется в Дейкстре.
- **Динамическое программирование (DP)** — метод оптимизации с хранением промежуточных результатов, лежит в основе Беллман–Форда и Флойда–Уоршелла.
- **Псевдограф, мультиграф** — обобщения графа: мультиграф допускает кратные рёбра, псевдограф — ещё и петли.

## 3. Основное содержание

### 3.1. Структура графа и базовые операции

#### Определение / Механизм

Граф представляется двумя структурами:

```c
// Узел графа
typedef struct {
    int node_id;       // Уникальный идентификатор узла
    Vector* neighbors; // Вектор соседей (siblings)
} Node;

// Граф
typedef struct {
    Vector* nodes;     // Вектор всех узлов
} Graph;
```

**Важно:** вектор `neighbors` хранит **идентификаторы** соседей (`unsigned int`), а не указатели на узлы. Это экономит память и позволяет получать узел по ID за $O(1)$ при использовании массива.

#### Создание графа

```c
Graph* createGraph() {
    Graph* graph = malloc(sizeof(Graph));
    if (!graph) {
        fprintf(stderr, "Memory for graph was not allocated :(\n");
        return NULL;
    }
    graph->nodes = createVector(sizeof(Node));
    return graph;
}
```

**Типичная ошибка:** после `malloc` всегда проверять, что указатель не `NULL`. В учебных целях допустим `exit`, но в production-коде нужно возвращать `NULL` или код ошибки — иначе при сбое в долгоживущем сервисе невозможно локализовать ошибку.

#### Создание узла

```c
int createNode(Graph* graph, unsigned int node_id) {
    if (!graph) {
        fprintf(stderr, "Graph is NULL\n");
        return -1;
    }
    Node* node = malloc(sizeof(Node));
    if (!node) return -1;
    node->node_id = node_id;
    node->neighbors = createVector(sizeof(Node*));
    appendVectorItem(graph->nodes, node);
    return 0;
}
```

**Инвариант:** при создании узла его `neighbors` — всегда пустой вектор (никогда не `NULL`). Это упрощает все дальнейшие операции: не нужно проверять `if (neighbors == NULL)`.

#### Поиск узла по ID

```c
Node* findNode(Graph* graph, unsigned int node_id) {
    for (int i = 0; i < graph->nodes->size; i++) {
        Node* node = getVectorItem(graph->nodes, i);
        if (node->node_id == node_id) {
            return node;
        }
    }
    fprintf(stderr, "Node with id %u was not found :(\n", node_id);
    return NULL;
}
```

**Сложность:** $O(V)$ — линейный поиск по всем вершинам. [дополнено] Для ускорения до $O(1)$ можно использовать хеш-таблицу `{node_id -> Node*}`. [/дополнено]

#### Добавление ребра

```c
int createEdge(Graph* graph, unsigned int node_id_from, unsigned int node_id_to) {
    if (!graph) return -1;
    Node* node_from = findNode(graph, node_id_from);
    Node* node_to   = findNode(graph, node_id_to);
    if (!node_from || !node_to) {
        fprintf(stderr, "Cannot create edge: node not found\n");
        return -1;
    }
    int flag = appendVectorItem(node_from->neighbors, node_to);
    if (flag != 0) {
        fprintf(stderr, "Cannot create edge from %u to %u\n",
                node_id_from, node_id_to);
        return -1;
    }
    return 0;
}
```

**Механизм:** ориентированное ребро $u \to v$ реализуется добавлением `node_to` в список `neighbors` узла `node_from`.

**Типичная ошибка:** при хранении указателей на узлы в векторе, который может перераспределять память (`realloc`), старые указатели становятся невалидными — dangling pointer. Решение: хранить в `neighbors` индексы или ID, а не прямые указатели.

#### Тестирование базовой реализации

```c
#include <assert.h>

// ...
Graph* graph = createGraph();
assert(graph != NULL);

createNode(graph, 1);
assert(graph->nodes->size == 1);

createNode(graph, 2);
assert(graph->nodes->size == 2);

int flag = createEdge(graph, 1, 2);
assert(flag == 0);

Node* node = findNode(graph, 1);
assert(node->node_id == 1);
assert(node->neighbors->size == 1);
```

**Правило:** перед написанием BFS/DFS всегда проверяется база — хотя бы создание, добавление и поиск. Иначе при отладке алгоритма невозможно понять, откуда ошибка.

### 3.2. Поиск в ширину (BFS)

#### Определение / Механизм

BFS (Breadth-First Search) обходит граф уровнями: сначала все соседи стартового узла, затем соседи соседей и т. д. Использует очередь (FIFO — First In, First Out).

#### Теория и сложность

- **Время:** $O(V + E)$ — каждый узел и каждое ребро обрабатываются ровно по одному разу.
- **Память:** $O(V)$ — в очереди одновременно хранится не более $V$ узлов.

#### Код

```c
void bfs(Graph* graph, unsigned int node_id_start) {
    GenericList* list = createList(sizeof(unsigned int));
    appendItem(list, &node_id_start);

    HashSetInt* visited = createHashSetInt();  // для отслеживания посещённых

    while (listLength(list) > 0) {
        unsigned int node_next_id = *(unsigned int*)popItem(list, 0);

        if (hashSetIntContains(visited, node_next_id)) continue;
        hashSetIntAdd(visited, &node_next_id);

        GNode* node_next = findNode(graph, node_next_id);
        printf("Visited node: %u\n", node_next_id);

        for (int i = 0; i < node_next->neighbors->size; i++) {
            GNode* node = getVectorItem(node_next->neighbors, i);
            appendItem(list, &node->node_id);
        }
    }
}
```

**Порядок обхода** для графа `1 -> 2, 1 -> 3, 2 -> 4`:
```
Visited node: 1
Visited node: 2
Visited node: 3
Visited node: 4
```

#### Интуиция

Представьте волну на воде от брошенного камня: сначала ближайший круг, потом следующий. BFS гарантирует нахождение **кратчайшего пути** (по числу рёбер) в невзвешенном графе.

#### Типичные ошибки

- **Нет отслеживания посещённых узлов:** при наличии циклов алгоритм зациклится. Для хранения посещённых узлов используется хеш-сет (`HashSetInt`), а не массив — ключи-ID хранить дешевле, чем целые структуры узлов.
- **Хранение узлов вместо ID в очереди:** вектор с узлами может перераспределить память; хранение ID безопаснее.

### 3.3. Поиск в глубину (DFS)

#### Определение / Механизм

DFS (Depth-First Search) идёт вглубь по одной ветке до тупика, затем возвращается (backtracking) и идёт по следующей. Реализуется рекурсивно или итеративно через явный стек (LIFO — Last In, First Out).

#### Теория и сложность

- **Время:** $O(V + E)$.
- **Память:** $O(V)$ — глубина рекурсии или размер стека.

#### Код — рекурсивная реализация

```c
void dfs(Graph* graph, unsigned int node_id_start) {
    GNode* node_next = findNode(graph, node_id_start);
    if (node_next->neighbors->size == 0) {
        return;
    }
    printf("Visited node: %u\n", node_id_start);
    for (int i = 0; i < node_next->neighbors->size; i++) {
        GNode* node = getVectorItem(node_next->neighbors, i);
        dfs(graph, node->node_id);
    }
}
```

**Порядок обхода** для того же графа:
```
Visited node: 1
Visited node: 2
Visited node: 4
Visited node: 3
```

**Проблема рекурсивной реализации:** при глубоких графах возможно переполнение стека вызовов (stack overflow).

#### Код — итеративная реализация через стек

```cpp
void dfs_without_recursion(Node* start) {
    std::stack<Node*> stack;
    std::vector<bool> visited(nodes.size(), false);
    stack.push(start);

    while (!stack.empty()) {
        Node* current = stack.top();
        stack.pop();

        if (!visited[current->node_id]) {
            std::cout << "Visited node: " << current->node_id << std::endl;
            visited[current->node_id] = true;

            for (auto neighbor : current->neighbors) {
                stack.push(neighbor);
            }
        }
    }
}
```

**Порядок обхода** с явным стеком (соседи добавляются в порядке списка, но извлекаются в обратном):
```
Visited node: 1
Visited node: 3
Visited node: 4
Visited node: 2
```

#### Интуиция

Рекурсия — это неявный стек вызовов. Явный стек позволяет управлять порядком обхода и избежать переполнения. Главное свойство стека: **положил последним — достал первым** (LIFO).

#### Типичные ошибки

- **Нет отслеживания посещённых:** в графе с циклами рекурсия или цикл будет выполняться бесконечно.
- **Путаница LIFO / FIFO:** стек — LIFO (DFS), очередь — FIFO (BFS). Замена одного на другое меняет алгоритм.

---

### 3.4. Сравнение BFS и DFS

| Свойство | BFS | DFS |
|---|---|---|
| Структура данных | Очередь (FIFO) | Стек (LIFO) / рекурсия |
| Порядок обхода | Уровнями (по ширине) | Вглубь по ветке |
| Кратчайший путь (невзв.) | Гарантирует | Не гарантирует |
| Память | $O(V)$ (широкие графы — много) | $O(V)$ (глубокие деревья — много) |
| Сложность | $O(V + E)$ | $O(V + E)$ |
| Применение | Кратч. путь, обход по уровням | Топосортировка, поиск циклов, SCС |

### 3.5. Топологическая сортировка

#### Определение / Механизм

**Топологическая сортировка** — определение линейного порядка обхода вершин в **ориентированном ациклическом графе (DAG)** такого, что для каждого ребра $u \to v$ узел $u$ стоит перед $v$.

Формально: для DAG $G = (V, E)$ топологический порядок — это биекция $f: V \to \{1, \dots, |V|\}$ такая, что $(u, v) \in E \Rightarrow f(u) < f(v)$.

[дополнено] Топологическая сортировка существует тогда и только тогда, когда граф является DAG (нет циклов). [/дополнено]

#### Теория и сложность

- **Время:** $O(V + E)$ — реализация на основе DFS.
- **Память:** $O(V)$ — хеш-сет посещённых + список пути.

#### Структура графа для топосортировки

```c
typedef struct {
    int node_id;
    Vector* neighbors; // "из" — ссылки на потомков
} Node;

typedef struct {
    Vector* nodes;
} Graph;
```

**Ключевой вопрос:** в поле `neighbors` хранятся ссылки **"из"** данного узла (то, на что он указывает) или **"в"** него (кто на него указывает)? В коде топосортировки лектора `neighbors` трактуются как «входящие» ссылки — узлы, которые **ведут в** данный, т. е. предшественники.

#### Код — вспомогательная итерация (рекурсивный DFS)

```c
void topologicalSortIter(Graph* graph, unsigned int node_id,
                         HashSetInt* nodes_exclude, GenericList* path) {
    GNode* node = findNode(graph, node_id);

    // Сначала обходим всех предшественников
    for (int i = 0; i < node->siblings->size; i++) {
        GNode* node_parent = *(GNode**)getVectorItem(node->siblings, i);
        if (!hashSetIntContains(nodes_exclude, node_parent->node_id)) {
            topologicalSortIter(graph, node_parent->node_id,
                                nodes_exclude, path);
        }
    }

    // После возврата из рекурсии добавляем текущий узел в путь
    appendItem(path, &node->node_id);
    hashSetIntAdd(nodes_exclude, &node->node_id);
}
```

**Ключевая идея:** узел добавляется в результирующий список **после** обработки всех своих предшественников — это обратный постпорядок DFS.

#### Код — обёртка

```c
void topologicalSort(Graph* graph) {
    GenericList* end_nodes = createList(sizeof(unsigned int));

    // Находим «конечные» узлы — те, у кого есть предшественники
    for (int i = 0; i < graph->nodes->size; i++) {
        GNode* node = *(GNode**)getVectorItem(graph->nodes, i);
        if (node->siblings->size > 0) {  // есть входящие рёбра
            appendItem(end_nodes, &node->node_id);
        }
    }

    HashSetInt* nodes_exclude = createHashSetInt();
    GenericList* path = createList(sizeof(unsigned int));

    for (int i = 0; i < listLength(end_nodes); i++) {
        unsigned int node_id = popItem(end_nodes, 0);
        topologicalSortIter(graph, node_id, nodes_exclude, path);
    }
    // path теперь содержит топологический порядок
}
```

#### Пример

Граф: рёбра `(3→1), (4→3), (1→4), (2→2)` — пример из презентации (нотация (from; index)):

```
Узлы: 1, 2, 3, 4
Если отобразить порядок: 1 → 3 → 2 → 4
```

#### Интуиция

Алгоритм аналогичен разгрузке зависимостей: сначала выполняются задачи, у которых нет зависимостей (источники), затем те, чьи зависимости уже выполнены. Практическое применение — **Celery** (планировщик задач Python): топосортировка pipeline из 100 задач даёт чёткий порядок постановки в очередь на N воркеров.

[дополнено] Топологический индекс узла действителен только для фиксированной структуры графа. При добавлении или удалении рёбер топологическая сортировка должна быть выполнена заново. [/дополнено]

#### Типичные ошибки

- **Граф с циклом:** топологическая сортировка неприменима; перед запуском нужно проверять ацикличность (например, с помощью DFS — детектировать «серые» вершины в стеке вызовов).
- **Путаница «из» / «в»:** неверная интерпретация направления рёбер в `neighbors` даст обратный порядок.

### 3.6. Алгоритм Дейкстры

#### Определение / Механизм

Алгоритм Дейкстры находит кратчайшие пути от одной исходной вершины до всех остальных в **взвешенном графе с неотрицательными весами рёбер**.

Для хранения весов рёбер структура узла расширяется: `neighbors` хранит пары `(Node*, int)` — сосед и вес ребра до него.

#### Теория и сложность

Зависит от реализации очереди с приоритетом:

| Реализация очереди | Время работы | Когда выгодно |
|---|---|---|
| Массив (линейный поиск) | $O(V^2 + E)$ | Плотные графы ($E \approx V^2$) |
| Список (линейный поиск) | $O(V^2 + E)$ | Плотные графы |
| Бинарная куча (min-heap) | $O(E \cdot \log V)$ | Разреженные графы ($E \ll V^2$) |
| Красно-чёрное дерево | $O(E \cdot \log V)$ | Нужна упорядоченность / диапазонные запросы |
| Куча Фибоначчи | $O(E + V \cdot \log V)$ | Теоретический оптимум, сложна в реализации |

#### Код (C++)

```cpp
std::vector<Node*> dijkstra(Node* start, Node* end) {
    std::unordered_map<int, int> distances;  // node_id -> минимальное расстояние
    for (Node* node : nodes)
        distances[node->node_id] = (node->node_id == start->node_id) ? 0 : -1;

    std::unordered_map<int, int> previous;   // node_id -> предыдущий node_id в пути

    // Очередь с приоритетом: (узел, расстояние), min по расстоянию
    std::priority_queue<std::pair<Node*, int>,
                        std::vector<std::pair<Node*, int>>,
                        /* компаратор по убыванию */ ...> pq;
    pq.push({start, 0});

    while (!pq.empty()) {
        auto [current_node, current_dist] = pq.top(); pq.pop();

        if (current_node == end) { /* восстановить путь через previous */ }

        for (auto [neighbor, weight] : current_node->neighbors) {
            int newDistance = distances[current_node->node_id] + weight;
            int neighborDistance = distances[neighbor->node_id];

            if (neighborDistance == -1 || newDistance < neighborDistance) {
                distances[neighbor->node_id] = newDistance;
                previous[neighbor->node_id] = current_node->node_id;
                pq.push({neighbor, newDistance});
            }
        }
    }
    return {};  // путь не найден
}
```

#### Интуиция

Жадный алгоритм: всегда расширять вершину с наименьшим известным расстоянием от источника. Визуально — «волна», которая распространяется быстрее там, где рёбра легче.

#### Проблема отрицательных весов

```
A --4--> B
A --2--> C
B --(-3)--> C
```

Дейкстра при $A \to C$: находит путь длиной 2 напрямую и «не знает», что через $B$ было бы $4 + (-3) = 1$. Причина: после извлечения вершины из очереди её расстояние считается финальным, но отрицательный вес может его уменьшить позже.

### 3.7. Алгоритм A\*

#### Определение / Механизм

A\* — модификация Дейкстры, которая добавляет **эвристику** $h(v)$ — оценку расстояния от текущей вершины до цели. Функция приоритета: $f(v) = g(v) + h(v)$, где $g(v)$ — фактическое расстояние от источника.

#### Примеры эвристик

- **Евклидово расстояние** в $N$-мерном пространстве: $h(v) = \sqrt{\sum_i (v_i - \text{goal}_i)^2}$
- **Формула хаверсина (Haversine)** — расстояние по поверхности сферы (для географических карт).

#### Сложность

При допустимой (не переоценивающей) эвристике A\* гарантирует оптимальный путь. В худшем случае — $O(E \cdot \log V)$, как Дейкстра, но на практике значительно быстрее.

### 3.8. Алгоритм Беллмана–Форда

#### Определение / Механизм

Алгоритм Беллмана–Форда находит кратчайшие пути от одной вершины в графе с **отрицательными весами рёбер** (но без отрицательных циклов).

#### Теория и сложность

- **Время:** $O(V \cdot E)$ — $V-1$ релаксаций всех рёбер.
- **Память:** $O(V)$ — массив расстояний.

#### Псевдокод

```python
# 1. Инициализация
dp = {v: 0 if v == source else infinity for v in V}

# 2. V-1 итераций релаксации
for v in range(len(V) - 1):
    for e in E:
        if dp[e.from] != infinity and dp[e.from] + e.weight < dp[e.to]:
            dp[e.to] = dp[e.from] + e.weight

# 3. Проверка на отрицательные циклы
for e in E:
    if dp[e.from] + e.weight < dp[e.to]:
        # Обнаружен отрицательный цикл!
```

#### Пример итераций для графа $A \xrightarrow{4} B, A \xrightarrow{2} C, B \xrightarrow{-3} C$

| Итерация | A | B | C |
|---|---|---|---|
| 0 | 0 | $\infty$ | $\infty$ |
| 1 | 0 | 4 | 2 |
| 2 | 0 | 4 | 1 |

Правильный ответ: $A \to B \to C = 1$.

#### Детектирование отрицательных циклов

После $V-1$ итераций выполняется ещё одна проверка: если хоть какое-то расстояние уменьшилось — в графе есть отрицательный цикл.

Граф с ребром $A \xrightarrow{-1} B$ и циклом $B \to C \to A$:

| Итерация | A | B | C |
|---|---|---|---|
| 0 | 0 | $\infty$ | $\infty$ |
| 1 | -1 | 4 | 2 |
| 2 | -1 | 3 | 0 |
| 3 | -2 | 3 | 0 |
| N | -2 | 2 | -1 |

После $V$-й итерации: расстояние уменьшилось $\Rightarrow$ отрицательный цикл.

#### Типичные ошибки

- Применение Дейкстры к графам с отрицательными весами: даёт неверный результат.
- Неправильная инициализация: расстояние до всех вершин кроме источника должно быть $+\infty$, а не 0.

### 3.9. Алгоритм Флойда–Уоршелла

#### Определение / Механизм

Алгоритм Флойда–Уоршелла находит **кратчайшие пути между всеми парами вершин** в графе (в том числе с отрицательными весами, но без отрицательных циклов). Работает на основе **матрицы смежности**.

#### Теория и сложность

- **Время:** $O(V^3)$.
- **Память:** $O(V^2)$ — матрицы `dp` и `path`.

#### Псевдокод

```python
# Инициализация: dp[u][v] = вес ребра (u,v) или +inf
# i - промежуточная вершина

for i in V:
    dp_cur = copy(dp)  # копия с весами для i-й итерации
    for u in V:
        for v in V:
            if dp_cur[u][i] + dp_cur[i][v] < dp_cur[u][v]:
                dp[u][v] = dp_cur[u][i] + dp_cur[i][v]
```

**Идея:** для каждой промежуточной вершины $i$ проверяем, не стал ли путь $u \to i \to v$ короче прямого $u \to v$.

#### Восстановление пути

```python
# Дополнительно храним матрицу предшественников
path = [[None] * V for _ in range(V)]

for i in V:
    dp_cur = copy(dp)
    for u in V:
        for v in V:
            if dp_cur[u][i] + dp_cur[i][v] < dp_cur[u][v]:
                dp[u][v] = dp_cur[u][i] + dp_cur[i][v]
                path[u][v] = path[u][i]  # путь через i
```

#### Интуиция

Алгоритм последовательно «улучшает» матрицу расстояний, разрешая использовать в пути первые $k$ вершин в качестве промежуточных. После $k$-й итерации $dp[u][v]$ — кратчайший путь, использующий только вершины $\{1, \dots, k\}$ как промежуточные.

## 4. Сравнительная таблица алгоритмов кратчайшего пути

| Алгоритм | Тип задачи | Отриц. веса | Отриц. циклы | Сложность | Память |
|---|---|---|---|---|---|
| Дейкстра (min-heap) | Один источник | Нет | Нет | $O(E \log V)$ | $O(V)$ |
| A\* | Один источник, одна цель | Нет | Нет | $O(E \log V)$* | $O(V)$ |
| Беллман–Форд | Один источник | Да | Детектирует | $O(V \cdot E)$ | $O(V)$ |
| Флойд–Уоршелл | Все пары | Да | Детектирует | $O(V^3)$ | $O(V^2)$ |

\* — в лучшем случае значительно быстрее при хорошей эвристике.

## 5. Справочный лист

### Структуры данных

- **BFS:** очередь (GenericList / `std::queue`), FIFO.
- **DFS:** стек (`std::stack`) или рекурсия, LIFO.
- **Дейкстра:** очередь с приоритетом (`std::priority_queue`, min-heap).
- **Посещённые узлы:** хеш-сет по ID — $O(1)$ добавление и поиск.
- **neighbors:** хранить ID (`unsigned int`), а не указатели на узлы.

### Сложности

| Операция | Структура | Сложность |
|---|---|---|
| Создание графа | — | $O(1)$ |
| Создание узла | — | $O(1)$ амортизированно |
| Поиск узла по ID | Массив/вектор | $O(V)$ |
| Поиск узла по ID | Хеш-таблица | $O(1)$ |
| BFS | Список смежности | $O(V + E)$ |
| DFS | Список смежности | $O(V + E)$ |
| Топосортировка | DFS | $O(V + E)$ |
| Дейкстра | Бинарная куча | $O(E \log V)$ |
| Беллман–Форд | — | $O(V \cdot E)$ |
| Флойд–Уоршелл | Матрица смежности | $O(V^3)$ |

### Ключевые правила

- FIFO (очередь) $\Rightarrow$ BFS.
- LIFO (стек) $\Rightarrow$ DFS.
- Отрицательные веса $\Rightarrow$ Беллман–Форд или Флойд–Уоршелл; Дейкстра не применима.
- Топосортировка $\Rightarrow$ только DAG; наличие цикла делает её невозможной.
- Топологический индекс вершины актуален только для зафиксированного графа.
- При `malloc` всегда проверять результат на `NULL`.
- При создании узла `neighbors` всегда инициализировать пустым вектором (не `NULL`).

## 6. Практические советы

1. **Проверяйте базу тестами перед написанием алгоритмов.** Несколько `assert`-ов на создание, добавление и поиск узлов займут минуту, но избавят от часов отладки BFS/DFS с неправильной структурой данных.

2. **Храните в очередях/стеках идентификаторы (int/unsigned int), а не указатели на узлы.** Вектор может перераспределить память, инвалидировав указатели. ID дешевле, и получение узла по ID из массива — $O(1)$.

3. **Всегда отслеживайте посещённые вершины.** Даже если вы «уверены», что граф ациклический — добавьте проверку. Используйте хеш-сет по ID, а не вектор булевых значений, если ID не являются плотным диапазоном $[0, V)$.

4. **Выбирайте алгоритм кратчайшего пути под задачу.** Дейкстра — быстро, но только для неотрицательных весов. Беллман–Форд — медленнее, но корректен при отрицательных рёбрах и умеет находить отрицательные циклы. A\* — для задач с пространственной эвристикой (карты, сетки). Флойд–Уоршелл — когда нужны пути между всеми парами.

5. **В production-коде не используйте `exit()` в низкоуровневых функциях.** Возвращайте коды ошибок или `NULL` — иначе при сбое в одном компоненте завалится весь сервис, и вы не сможете локализовать проблему.
