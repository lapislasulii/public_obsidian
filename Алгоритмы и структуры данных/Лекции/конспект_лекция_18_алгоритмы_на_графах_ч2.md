## 1. Обзор лекции

Лекция посвящена практическим аспектам реализации и анализа алгоритмов на графах. Рассматриваются вопросы представления графа в памяти: копирование структур, управление указателями, проблемы утечек памяти и несогласованности атрибутов. Основное содержание охватывает топологическую сортировку ориентированного ациклического графа (DAG), алгоритм Дейкстры для поиска кратчайшего пути (с вариантами реализации приоритетной очереди), алгоритм A* как расширение Дейкстры с эвристикой, а также алгоритмы Беллмана — Форда и Флойда — Уоршелла для работы с отрицательными весами. Лекция завершается обзором асимптотик и практических рекомендаций по выбору структур данных для каждого алгоритма.

## 2. Предварительные сведения

- **Граф** — множество вершин $V$ и рёбер $E$; обозначается $G = (V, E)$.
- **Ориентированный граф (орграф)** — рёбра имеют направление; различают «входящие» и «исходящие».
- **DAG (Directed Acyclic Graph)** — ориентированный граф без циклов; необходимое условие для топологической сортировки.
- **BFS (Breadth-First Search)** — обход графа в ширину, использует очередь.
- **DFS (Depth-First Search)** — обход графа в глубину, использует рекурсию или явный стек.
- **Список смежности** — для каждой вершины хранится список её соседей.
- **Матрица смежности** — двумерная матрица $V \times V$, ячейка содержит вес или флаг наличия ребра.
- **Динамическое программирование (DP)** — оптимизация рекурсии через сохранение промежуточных результатов.
- **Эвристика** — вспомогательная оценочная функция, ускоряющая поиск в пространстве состояний.
- **Амортизационная сложность** — усреднённая стоимость операции по серии вызовов.
- **Куча (heap, min-heap)** — структура данных для эффективного извлечения минимального элемента за $O(\log n)$.
- **Красно-чёрное дерево (RB-tree)** — самобалансирующееся BST, поиск/вставка/удаление за $O(\log n)$.
- **Куча Фибоначчи** — амортизированно $O(1)$ вставка и уменьшение ключа, $O(\log n)$ извлечение минимума.

## 3. Основное содержание

### 3.1. Представление графа в памяти: управление указателями

#### Определение и механизм

В реализации используется структура узла `GNode`, содержащая `node_id` (целое) и указатель `neighbors` (или `siblings`) на вектор соседних узлов. Граф (`Graph`) хранит вектор указателей на такие узлы.

```c
typedef struct {
    int node_id;       // идентификатор узла
    Vector* neighbors; // указатель на вектор соседей
} GNode;

typedef struct {
    Vector* nodes;     // вектор указателей на узлы
} Graph;
```

#### Проблема 1: мелкое копирование (shallow copy)

Когда узел добавляется в вектор через `memcpy`, копируется пара `(node_id, указатель на вектор соседей)`. Размер структуры — 16 байт (8 байт `int` с выравниванием + 8 байт указатель). Копируется именно указатель, а не содержимое вектора. Итог — несколько записей в разных местах графа ссылаются на **один и тот же** вектор соседей.

**Следствия:**
- Изменение атрибута (например, `color`) в одном узле не отражается в копии — несогласованность данных.
- При удалении узла из графа требуется обойти **все** остальные узлы и их векторы соседей, чтобы удалить ссылку: сложность $O(V + E)$ на каждое удаление.

#### Проблема 2: утечка памяти

После копирования указателя в вектор оригинальный локальный `GNode` уничтожается, но **выделенная под него память не освобождается**. Это классическая утечка памяти (memory leak).

#### Решение: хранение указателей на узлы вместо значений

Вместо хранения самого узла в векторе хранится **указатель** на узел. Размер каждого элемента вектора сокращается с 16 до 8 байт (один указатель). Все ссылки указывают на **оригинальный** узел, обновление атрибутов видно повсеместно.

[дополнено]
Это паттерн «единственный владелец» (single ownership): граф владеет узлами, а рёбра — лишь не-владеющие (non-owning) указатели. В C++ это соответствует использованию `std::unique_ptr<GNode>` в векторе узлов и `GNode*` в списках соседей.
[/дополнено]

**Новая проблема:** при обращении к элементу вектора получается `GNode**` (указатель на указатель). Необходимо двойное разыменование:

```c
GNode* node = *(GNode**)getVectorItem(graph->nodes, i);
```

#### Типичные ошибки

- Забыть двойное разыменование при итерации по вектору узлов.
- Не освобождать память под вектор соседей при удалении узла.
- Хранить `node_id` в качестве ключей, но не переходить на хэш-таблицу — получение узла по `id` стоит $O(V)$ вместо $O(1)$.

[дополнено]
Для получения узла по `id` за $O(1)$ следует хранить граф не как `Vector* nodes`, а как `HashMap<int, GNode*>`. Тогда `findNode` выполняется за $O(1)$ амортизированно.
[/дополнено]

### 3.2. BFS и DFS — повторение и сравнение

#### BFS (поиск в ширину)

**Структура:** очередь (FIFO).  
**Порядок обхода:** уровень за уровнем от стартовой вершины.

```c
void bfs(Graph* graph, unsigned int node_id_start) {
    GenericList* list = createList(sizeof(unsigned int));
    appendItem(list, &node_id_start);
    while (listLength(list) > 0) {
        unsigned int node_next_id = *(unsigned int*)popItem(list, 0);
        GNode* node_next = findNode(graph, node_next_id);
        for (int i = 0; i < node_next->siblings->size; i++) {
            GNode* node = getVectorItem(node_next->siblings, i);
            appendItem(list, &node->node_id);
        }
    }
}
```

Вывод для графа `1 -> 2 -> 4`, `1 -> 3`:
```
Visited node: 1
Visited node: 2
Visited node: 3
Visited node: 4
```

#### DFS (поиск в глубину, рекурсивный)

**Структура:** стек вызовов (рекурсия).  
**Порядок обхода:** углубляется по первому соседу до листа.

```c
void dfs(Graph* graph, unsigned int node_id_start) {
    GNode* node_next = findNode(graph, node_id_start);
    if (node_next->siblings->size == 0) {
        return;
    }
    for (int i = 0; i < node_next->siblings->size; i++) {
        GNode* node = getVectorItem(node_next->siblings, i);
        dfs(graph, node->node_id);
    }
}
```

Вывод:
```
Visited node: 1
Visited node: 2
Visited node: 4
Visited node: 3
```

**Тяга (проблема) текущей рекурсивной реализации DFS:** нет проверки посещённости — граф с циклами приведёт к бесконечной рекурсии.

#### DFS без рекурсии (через явный стек)

```c
void dfs_without_recursion(Node* start) {
    std::stack<Node*> stack;
    std::vector<bool> visited(nodes.size(), false);
    stack.push(start);
    while (!stack.empty()) {
        Node* current = stack.top();
        stack.pop();
        if (!visited[current->node_id]) {
            std::cout << "Visited node: " << current->node_id << std::endl;
            visited[current->node_id] = true;
            for (auto neighbor : current->neighbors) {
                stack.push(neighbor);
            }
        }
    }
}
```

[дополнено]
Порядок обхода при итеративном DFS может отличаться от рекурсивного из-за порядка добавления соседей в стек. Это не ошибка, а нормальная особенность реализации.
[/дополнено]

**Интуиция:** DFS напоминает блуждание по лабиринту с нитью Ариадны — всегда идёшь вперёд до тупика, затем возвращаешься. BFS — концентрические волны от точки бросания камня в воду.

### 3.3. Топологическая сортировка

#### Определение

**Топологическая сортировка** — линейное упорядочение вершин ориентированного ациклического графа (DAG) такое, что для каждого ребра $(u \to v)$ вершина $u$ стоит перед $v$ в этом порядке.

- Работает **только** на DAG (ориентированных ациклических графах).
- На неориентированных или циклических графах не определена.
- Если граф несвязный — алгоритм всё равно работает, обрабатывая каждую компоненту.

#### Механизм (DFS-подход через родителей)

Ключевая идея: хранить для каждого узла список его **родителей** (т.е. вершины, из которых ведут рёбра), а не детей. Алгоритм:

1. Найти все «конечные» узлы (у которых есть исходящие рёбра, т.е. родители в терминах хранения).
2. Для каждого такого узла запустить рекурсивную функцию.
3. В рекурсии: сначала обработать всех родителей (рекурсивно), затем — добавить текущий узел в путь и отметить его как посещённый.

Четыре логических блока функции:
```
[базовый случай] → [предпроцессинг] → [рекурсия по родителям] → [постпроцессинг: добавить узел в путь]
```

**Базовый случай** — нет непосещённых родителей (автоматически, без явного `if`).  
**Постпроцессинг** — добавить узел в список `path` и в `nodes_exclude`.

#### Код

```c
void topologicalSortIter(Graph* graph, unsigned int node_id,
                         HashSetInt* nodes_exclude, GenericList* path) {
    GNode* node = findNode(graph, node_id);
    for (int i = 0; i < node->siblings->size; i++) {
        GNode* node_parent = *(GNode**)getVectorItem(node->siblings, i);
        if (!hashSetIntContains(nodes_exclude, node_parent->node_id)) {
            topologicalSortIter(graph, node_parent->node_id,
                                nodes_exclude, path);
        }
    }
    appendItem(path, &node->node_id);
    hashSetIntAdd(nodes_exclude, &node->node_id);
}
```

Обёртка (запуск для всех конечных узлов):

```c
void topologicalSort(Graph* graph) {
    GenericList* end_nodes = createList(sizeof(unsigned int));
    for (int i = 0; i < graph->nodes->size; i++) {
        GNode* node = *(GNode**)getVectorItem(graph->nodes, i);
        if (node->siblings->size > 0) {  // "В" — есть родители
            appendItem(end_nodes, &node->node_id);
        }
    }
    GenericList* nodes_exclude = createList(sizeof(unsigned int));
    GenericList* path = createList(sizeof(unsigned int));
    for (int i = 0; i < listLength(end_nodes); i++) {
        unsigned int node_id = popItem(end_nodes, 0);
        topologicalSortIter(graph, node_id, nodes_exclude, path);
    }
}
```

#### Асимптотика

- Каждая вершина обрабатывается ровно один раз: $O(V)$.
- Каждое ребро просматривается ровно один раз: $O(E)$.
- Итого: $O(V + E)$.
- При использовании списка для `nodes_exclude` вместо `HashSet` стоимость проверки вырастает до $O(V)$, а общая — до $O(V \cdot (V + E))$.

#### Хранение пути

- **Связный список** — дешёвое добавление в конец, $O(1)$.
- **Вектор (с заранее выделенной памятью $V$)** — $O(1)$ без амортизации, удобен для индексного доступа.

#### Вспомогательная структура `HashSetInt`

Рекомендуется реализовать собственный `HashSetInt` (хэш-множество целых чисел), так как большинство значений в задачах на графах — целые идентификаторы. Проверка принадлежности — $O(1)$ амортизированно.

#### Интуиция

Алгоритм похож на выполнение задач с зависимостями: прежде чем выполнить задачу, нужно завершить все задачи-предшественники. Порядок добавления в путь — обратный порядку обхода (постпроцессинг).

#### Типичные ошибки

- Хранить `children` вместо `parents` — алгоритм не сможет подняться к источникам графа.
- Не помечать узлы как посещённые — бесконечная рекурсия при наличии «алмазных» структур (один узел имеет несколько потомков, ведущих к нему через разные пути).
- Применять алгоритм к графу с циклами — бесконечная рекурсия.
- Добавлять узел в `path` **до** рекурсии (предпроцессинг) вместо **после** (постпроцессинг) — неправильный порядок.

### 3.4. Алгоритм Дейкстры

#### Определение

**Алгоритм Дейкстры** — жадный алгоритм нахождения кратчайших путей от одной источниковой вершины до всех остальных в графе с **неотрицательными** весами рёбер.

#### Добавление весов к структуре графа

Исходная структура хранит только `node_id` в векторе соседей. Для Дейкстры нужны веса. Варианты:

1. **Пара (id, weight) как элемент вектора:** каждый элемент вектора соседей — структура из 8 байт `id` + 8 байт `weight` = 16 байт. Просто и кэш-дружественно.

2. **Указатель на структуру ребра:** каждый элемент вектора — указатель (8 байт) на структуру `Edge { int from_id; int to_id; int weight; ... }`. Позволяет хранить произвольные атрибуты ребра (тип, цвет, стоимость), но добавляет косвенность.

3. **Хэш-таблица рёбер:** `HashMap<child_id, List<(parent_id, weight)>>`. Ключ — `child_id`, значение — список пар. При одном ребёнке от нескольких родителей: значение — список, не скаляр. Даёт $O(1)$ поиск всех рёбер, входящих в вершину.

[дополнено]
Выбор структуры хранения рёбер критически влияет на асимптотику Дейкстры. Для разреженных графов ($E \ll V^2$) оптимален список смежности. Для плотных графов ($E \approx V^2$) матрица смежности даёт более кэш-дружественный доступ.
[/дополнено]

#### Механизм алгоритма

```python
def dijkstra(graph, start, end):
    distances = {node: 0 if node == start else INF for node in graph}
    previous = {}
    pq = PriorityQueue()   # минимальная куча по расстоянию
    pq.push((start, 0))

    while not pq.empty():
        current, dist = pq.pop_min()
        if current == end:
            break
        for neighbor, weight in current.neighbors:
            new_dist = distances[current] + weight
            if new_dist < distances[neighbor]:
                distances[neighbor] = new_dist
                previous[neighbor] = current
                pq.push((neighbor, new_dist))

    return reconstruct_path(previous, end)
```

#### Код (C++, список смежности)

```c
std::vector<Node*> dijkstra(Node* start, Node* end) {
    std::unordered_map<int, int> distances;
    for (Node* node : nodes)
        distances[node->node_id] = (node->node_id == start->node_id) ? 0 : -1;

    std::unordered_map<int, int> previous;
    std::priority_queue<std::pair<Node*, int>, ...> pq;
    pq.push(std::pair(start, 0));

    while (!pq.empty()) {
        std::pair<Node*, int> current = pq.top(); pq.pop();
        if (current.first == end) { /* восстановить путь */ }
        for (std::pair<Node*, int> neighbor : current.first->neighbors) {
            int newDistance = distances[current.first->node_id] + neighbor.second;
            int neighborDistance = distances[neighbor.first->node_id];
            if (neighborDistance == -1 || newDistance < neighborDistance) {
                // Обновить distances, previous; добавить neighbor в pq
            }
        }
    }
    return {}; // путь не найден
}
```

#### Ключевые операции и их стоимость

На каждой итерации выполняются две дорогостоящие операции:
1. **Извлечение узла с минимальным расстоянием** из очереди.
2. **Обновление расстояния** до соседей (decrease-key или повторная вставка).

#### Интуиция

Дейкстра — жадный алгоритм. Он всегда берёт ближайшую непосещённую вершину и «фиксирует» её расстояние. Это работает только при неотрицательных весах: отрицательное ребро могло бы «улучшить» уже зафиксированное расстояние.

#### Типичные ошибки

- Применять к графу с отрицательными весами.
- Не проверять, посещена ли вершина (при повторной вставке в очередь).
- Использовать `distances[v] = -1` для обозначения бесконечности — при сравнении `newDistance < -1` результат всегда `false`, если `newDistance > 0`. Лучше использовать `INT_MAX` или специальный флаг.

### 3.5. Алгоритм A* (A-Star)

#### Определение

**A\*** — информированный алгоритм поиска кратчайшего пути, расширение Дейкстры. В приоритетной очереди используется не чистое расстояние $g(v)$, а сумма $f(v) = g(v) + h(v)$, где $h(v)$ — **эвристика** (оценка расстояния от $v$ до цели).

#### Что такое эвристика

**Эвристика** $h(v)$ — вспомогательная функция, оценивающая «расстояние» от текущей вершины до целевой. Требование к корректности: $h(v) \leq \text{реальное расстояние}(v, \text{цель})$ — так называемая допустимая (admissible) эвристика.

Типичные эвристики:
- **Евклидово расстояние** — для географических карт с координатами.
- **Формула Хаверсина (Haversine)** — геодезическое расстояние по сфере Земли между двумя точками по широте и долготе.
- **Манхэттенское расстояние** — для сеточных карт.

#### Механизм

```
f(v) = g(v) + h(v)
```

- $g(v)$ — фактическое расстояние от старта до $v$ (как в Дейкстре).
- $h(v)$ — эвристическая оценка от $v$ до цели.

Алгоритм аналогичен Дейкстре, но очередь сортируется по $f(v)$ вместо $g(v)$.

#### Когда A* лучше Дейкстры

**Пример:** граф с двумя путями. Прямое ребро весом 100 и цепочка из 200 рёбер весом 1 каждое. Дейкстра (жадная) сначала пойдёт по коротким рёбрам цепочки. A* с эвристикой (например, евклидово расстояние до цели) «поймёт», что прямое ребро ведёт ближе к цели, и исследует его первым.

**Проблема эвристики:** если между стартом и целью стоит непреодолимая преграда (барьер узлов), A* будет «тянуться» к цели и может застрять, обходя барьер неэффективно. Дейкстра в этом случае ведёт себя более равномерно.

[дополнено]
Если эвристика равна нулю ($h(v) = 0$ для всех $v$), A* вырождается в Дейкстру. Если эвристика переоценивает расстояние ($h(v) > \text{реальное}$), алгоритм перестаёт быть оптимальным (может найти не кратчайший путь).
[/дополнено]

#### Применение: поиск альтернативных маршрутов

При необходимости найти альтернативные маршруты (помимо основного, найденного Дейкстрой) можно использовать A* с модифицированной эвристикой, учитывающей не только расстояние до цели, но и «удалённость» от основного маршрута.

### 3.6. Алгоритм Беллмана — Форда

#### Определение

**Алгоритм Беллмана — Форда** — алгоритм нахождения кратчайших путей от одной вершины до всех остальных в графе, допускающем **отрицательные веса** рёбер. Дополнительно обнаруживает **отрицательные циклы**.

#### Причина неприменимости Дейкстры к отрицательным весам

Пример:
```
A --4--> B --(-3)--> C
A -----------2-----------> C
```
Дейкстра зафиксирует расстояние $A \to C = 2$ (прямое ребро) раньше, чем исследует путь $A \to B \to C = 4 + (-3) = 1$. Жадность не позволяет «пересмотреть» зафиксированное расстояние.

#### Механизм

1. Создать таблицу DP размером $1 \times V$: $dp[v] = 0$ для старта, $dp[v] = \infty$ для остальных.
2. Выполнить $V - 1$ итераций релаксации всех рёбер:

```python
for v in range(V):          # V итераций
    for e in E:             # по всем рёбрам
        if dp[e.from] != INF and dp[e.from] + e.weight < dp[e.to]:
            dp[e.to] = dp[e.from] + e.weight
```

3. Выполнить ещё одну (V-ю) итерацию: если какое-либо расстояние уменьшилось — в графе есть **отрицательный цикл**.

#### Пример заполнения таблицы

Граф: $A \xrightarrow{4} B$, $B \xrightarrow{-3} C$, $A \xrightarrow{2} C$

| Итерация | A | B   | C   |
|----------|---|-----|-----|
| 0        | 0 | INF | INF |
| 1        | 0 | 4   | 2   |
| 2        | 0 | 4   | 1   |
| 3        | 0 | 4   | 1   |

#### Пример с отрицательным циклом

Граф: $A \xrightarrow{4} B$, $B \xrightarrow{-3} C$, $C \xrightarrow{-1} A$

| Итерация | A  | B | C  |
|----------|----|---|----|
| 0        | 0  | INF | INF |
| 1        | -1 | 4   | 2   |
| 2        | -1 | 3   | 0   |
| N        | -2 | 2  | -1  |

После $V$-й итерации расстояния продолжают уменьшаться — цикл обнаружен.

#### Асимптотика

$O(V \cdot E)$

#### Интуиция

На каждой итерации алгоритм «распространяет» информацию о кратчайших путях на один шаг дальше от источника. За $V - 1$ итерацию гарантированно найдены все кратчайшие пути (путь длиннее $V - 1$ ребра в простом графе невозможен).

#### Типичные ошибки

- Выполнить только $V - 1$ итерацию и не проверить наличие отрицательного цикла.
- Использовать Беллмана — Форда там, где Дейкстра применима — нет смысла в повышении сложности.

### 3.7. Алгоритм Флойда — Уоршелла

#### Определение

**Алгоритм Флойда — Уоршелла** находит кратчайшие пути между **всеми парами вершин** графа. Работает с матрицей смежности. Допускает отрицательные веса, обнаруживает отрицательные циклы (если $dp[v][v] < 0$).

#### Механизм

```python
# dp[u][v] = вес ребра (u,v) или +INF, если ребра нет; dp[v][v] = 0
for i in V:                        # промежуточная вершина
    dp_cur = copy(dp)              # копия матрицы
    for u in V:
        for v in V:
            if dp_cur[u][i] + dp_cur[i][v] < dp_cur[u][v]:
                dp[u][v] = dp_cur[u][i] + dp_cur[i][v]
```

Параметр $i$ — промежуточная вершина; алгоритм последовательно проверяет, улучшается ли путь $u \to v$ при добавлении вершины $i$ как промежуточной.

#### Восстановление пути

```python
# path[u][v] = следующая вершина после u на пути к v
for i in V:
    dp_cur = copy(dp)
    for u in V:
        for v in V:
            if dp_cur[u][i] + dp_cur[i][v] < dp_cur[u][v]:
                dp[u][v] = dp_cur[u][i] + dp_cur[i][v]
                path[u][v] = path[u][i]  # следующий шаг u->v через i
```

#### Асимптотика

$O(V^3)$ по времени, $O(V^2)$ по памяти.

#### Когда применять

Когда нужны кратчайшие пути **между всеми парами** вершин. Для одной пары достаточно Дейкстры или Беллмана — Форда.

## 5. Справочный лист

**Топологическая сортировка:**
- Применима только к DAG.
- Асимптотика: $O(V + E)$ при `HashSet` для посещённых.
- Хранить рёбра как `parents` (входящие), а не `children`.
- Постпроцессинг (добавление в путь) — после рекурсии, не до.
- Для ориентированного графа с несколькими компонентами: запускать для каждой.

**Дейкстра:**
- Только неотрицательные веса.
- Жадный выбор: всегда берём вершину с минимальным $dist$.
- При `min-heap`: $O(E \log V)$.
- При линейном поиске: $O(V^2)$ — выгодно для плотных графов.
- Хранить `previous` для восстановления пути.

**A\*:**
- $f(v) = g(v) + h(v)$, где $g$ — факт. расстояние, $h$ — эвристика.
- Эвристика должна быть допустимой: $h(v) \leq \text{real\_dist}(v, \text{goal})$.
- При $h = 0$ вырождается в Дейкстру.
- Типичная эвристика для карт: Хаверсин или Евклид.

**Беллман — Форд:**
- Допускает отрицательные веса, обнаруживает отрицательные циклы.
- Асимптотика: $O(V \cdot E)$.
- Инициализация: $dp[\text{start}] = 0$, остальные $= \infty$.
- Выполнить $V - 1$ итераций релаксации + 1 проверочную.

**Флойд — Уоршелл:**
- Все пары кратчайших путей.
- Асимптотика: $O(V^3)$ время, $O(V^2)$ память.
- Работает с матрицей смежности.
- При $dp[v][v] < 0$ — обнаружен отрицательный цикл.

**Хранение графа:**
- Список смежности: $O(V + E)$ память, эффективен для разреженных графов.
- Матрица смежности: $O(V^2)$ память, $O(1)$ проверка наличия ребра.
- Хранить указатели на узлы (не сами узлы) для корректности атрибутов.
- `HashSetInt` для посещённых — $O(1)$ проверка.

**Разреженный граф:** $E \ll V^2$ — выгоден список смежности, min-heap.  
**Плотный граф:** $E \approx V^2$ — выгодна матрица, массив для очереди.

## 6. Практические советы

1. **Выбирайте структуру хранения рёбер под задачу.** Для Дейкстры обходите в направлении $\text{parent} \to \text{children}$ — храните исходящие рёбра. Для топологической сортировки — входящие (`parents`). Путаница направлений — самая частая ошибка.

2. **Никогда не применяйте Дейкстру к графам с отрицательными весами.** Используйте Беллмана — Форда или проверьте граф на отрицательные рёбра перед запуском.

3. **Реализуйте `HashSetInt` как вспомогательную структуру.** Большинство задач на графах оперирует целыми `node_id`. Хэш-множество даёт $O(1)$ проверку посещённости и ускоряет топологическую сортировку, Дейкстру и DFS.

4. **Храните узлы как указатели, а не значения.** Вектор `Vector<GNode*>` вместо `Vector<GNode>` исключает проблемы несогласованности атрибутов и упрощает удаление узлов.

5. **При выборе реализации приоритетной очереди для Дейкстры учитывайте плотность графа.** Для разреженных графов (`E ≪ V²`) min-heap даёт $O(E \log V)$; для плотных — простой массив с линейным поиском минимума ($O(V^2)$) может быть не хуже, а реализация проще. Куча Фибоначчи теоретически оптимальна ($O(E + V \log V)$), но сложна в реализации — используйте только при явной необходимости.
