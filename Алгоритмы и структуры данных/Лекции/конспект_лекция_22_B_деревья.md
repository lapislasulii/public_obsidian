## 1. Обзор лекции

Лекция охватывает три взаимосвязанные структуры данных: бинарные деревья поиска (BST), красно-чёрные деревья (Red-Black Tree) и B-деревья. Сначала разбирается базовая структура BST с её операциями поиска, вставки, удаления и обходов, а также принципиальная проблема несбалансированности. Затем вводятся красно-чёрные деревья как механизм гарантированной балансировки с помощью инвариантов окраски и операций поворота. Наконец, рассматриваются B-деревья как обобщение идей балансировки для систем с дисковым хранилищем — в частности, для индексов баз данных. Все три структуры обеспечивают операции поиска, вставки и удаления за $O(\log n)$ при правильной реализации, но достигают этого разными способами.

## 2. Предварительные сведения

- **Рекурсия и рекуррентные соотношения** — дерево обходится рекурсивно; сложность задаётся формулой $T(n) = aT(n/b) + f(n)$.
- **Указатели и связные структуры** — каждый узел дерева содержит значение и указатели на дочерние узлы и родителя.
- **Понятие высоты дерева** — высота $h$ определяет сложность операций: $O(h)$.
- **Логарифм по основанию 2** — в идеально сбалансированном дереве $h = \lfloor \log_2 n \rfloor$.
- **Бинарный поиск** — идея «делить пространство пополам» лежит в основе BST.
- **Базы данных и индексы** — дерево-структуры используются для оптимизации запросов по диапазону.

## 3. Основное содержание

### 3.1 Бинарное дерево поиска (Binary Search Tree, BST)

#### Определение и механизм

Бинарное дерево поиска — это дерево, в котором для каждого узла $x$ выполняется:

- все ключи в левом поддереве $< x.\text{key}$,
- все ключи в правом поддереве $> x.\text{key}$.

Структура узла:

```c
struct Node {
    int key;
    int count;       // для дубликатов
    Node *left;
    Node *right;
    Node *parent;
};
```

[дополнено]Если допускаются повторяющиеся ключи, хранить дубликаты в отдельных узлах нецелесообразно. Вместо этого к каждому узлу добавляют счётчик `count`. При вставке существующего ключа счётчик увеличивается, при удалении — уменьшается; реальное удаление узла происходит только при `count == 0`.[/дополнено]

#### Практическая мотивация: индексы в базах данных

Представьте CSV-файл (базу данных) с миллиардом строк. Строки упорядочены по `id`, поэтому по `id` можно делать бинарный поиск за $O(\log n)$. Но если нужно искать по другому столбцу (например, по цене), прямой бинарный поиск невозможен. Решение — **индекс**: отдельная структура, хранящая пары `(значение_поля, id_строки)` в порядке значений поля. BST является одной из базовых реализаций такого индекса.

Проблема плоской таблицы-индекса: вставка нового элемента требует сдвига всех последующих элементов — $O(n)$. BST позволяет делать вставку за $O(h)$.

#### Теория и сложность

| Операция | Лучший случай | Средний случай | Худший случай |
|----------|:---:|:---:|:---:|
| Поиск | $O(\log n)$ | $O(\log n)$ | $O(n)$ |
| Вставка | $O(\log n)$ | $O(\log n)$ | $O(n)$ |
| Удаление | $O(\log n)$ | $O(\log n)$ | $O(n)$ |
| Min / Max | $O(\log n)$ | $O(\log n)$ | $O(n)$ |
| Predecessor / Successor | $O(\log n)$ | $O(\log n)$ | $O(n)$ |

[дополнено]Худший случай $O(n)$ достигается при вставке уже отсортированной последовательности: дерево вырождается в связный список («бамбук»). Математическое ожидание высоты случайно построенного BST на $n$ различных ключах равно $\Theta(\log n)$: чем ближе порядок вставок к случайному, тем меньше высота.[/дополнено]

#### Обходы дерева

Все обходы выполняются за $\Theta(n)$ — каждый узел посещается ровно один раз.

```python
# Прямой обход (Pre-order): корень -> левое -> правое
def preorder(node):
    if node is None:
        return
    print(node.key)      # действие ДО рекурсии
    preorder(node.left)
    preorder(node.right)

# Центральный обход (In-order): левое -> корень -> правое
# Даёт ключи в ВОЗРАСТАЮЩЕМ порядке — главное свойство!
def inorder(node):
    if node is None:
        return
    inorder(node.left)
    print(node.key)      # действие В СЕРЕДИНЕ
    inorder(node.right)

# Обратный обход (Post-order): левое -> правое -> корень
def postorder(node):
    if node is None:
        return
    postorder(node.left)
    postorder(node.right)
    print(node.key)      # действие ПОСЛЕ рекурсии
```

**Интуиция:** In-order обход идёт от самого левого узла к самому правому, порождая отсортированную последовательность. Pre-order начинает с корня и идёт «слева направо». Post-order используется при удалении дерева из памяти (сначала освобождаем детей, потом родителя).

#### Поиск минимума, максимума

```python
def find_min(node):
    while node.left is not None:
        node = node.left
    return node

def find_max(node):
    while node.right is not None:
        node = node.right
    return node
```

Сложность: $O(h)$, где $h$ — высота дерева.

#### Predecessor и Successor (предыдущий и следующий элемент)

- **Successor(x):** если у $x$ есть правое поддерево — найти минимум правого поддерева. Иначе — идти вверх, пока не окажемся левым потомком.
- **Predecessor(x):** если у $x$ есть левое поддерево — найти максимум левого поддерева. Иначе — идти вверх, пока не окажемся правым потомком.

[дополнено]Пример: в дереве со значениями `{2, 5, 6, 7, 8, 13, 15}` predecessor числа 15 — это 13 (максимум левого поддерева относительно 15). Если вставить 14, оно встанет правым потомком 13 и станет новым predecessor числа 15.[/дополнено]

#### Вставка

```python
def insert(root, key):
    new_node = Node(key, color=RED)
    parent = None
    current = root
    while current is not None:
        parent = current
        if key < current.key:
            current = current.left
        elif key > current.key:
            current = current.right
        else:
            current.count += 1  # дубликат
            return root
    new_node.parent = parent
    if parent is None:
        root = new_node
    elif key < parent.key:
        parent.left = new_node
    else:
        parent.right = new_node
    return root
```

Сложность: $O(h)$.

#### Удаление (четыре случая)

При удалении узла $z$ рассматриваются четыре ситуации:

1. **У $z$ нет левого потомка** — правый потомок (или `None`) занимает место $z$.
2. **У $z$ есть только левый потомок** — левый потомок занимает место $z$.
3. **У $z$ есть оба потомка, и successor $y$ является непосредственным правым потомком** — $y$ поднимается на место $z$.
4. **У $z$ есть оба потомка, и successor $y$ находится глубже** — сначала $y$ извлекается со своего места (его правое поддерево занимает его место), затем $y$ ставится на место $z$.

[дополнено]Во всех случаях используется вспомогательная процедура `transplant(u, v)`, которая заменяет поддерево с корнем $u$ поддеревом с корнем $v$. Эта абстракция упрощает код и уменьшает количество операций с указателями.[/дополнено]

**Типичные ошибки:**
- Забыть обновить указатель `parent` при `transplant`.
- Не обработать случай, когда удаляемый узел является корнем дерева.
- Перепутать successor и predecessor при удалении узла с двумя потомками.

### 3.2 Красно-чёрное дерево (Red-Black Tree, RBT)

#### Определение и механизм

Красно-чёрное дерево — это BST, в котором каждый узел окрашен в красный или чёрный цвет, а набор инвариантов гарантирует, что дерево остаётся приблизительно сбалансированным.

Структура узла:

```c
struct RBNode {
    int key;
    char color;       // RED или BLACK
    RBNode *left;
    RBNode *right;
    RBNode *parent;
};
```

#### Пять инвариантов красно-чёрного дерева

1. Каждый узел окрашен либо в красный, либо в чёрный цвет.
2. **Корень всегда чёрный.**
3. Все листья (`NULL`-указатели, фиктивные `NIL`-узлы) — чёрные.
4. **Если узел красный, оба его потомка чёрны** (два красных узла не могут идти подряд).
5. **Для любого узла все простые пути от него до листьев содержат одинаковое количество чёрных узлов** (черная высота).

[дополнено]В реализации Кормена вместо `NULL`-указателей используется единственный сторожевой объект `T.nil` — чёрный «листовой» узел. Это упрощает граничные проверки: все листья и родитель корня указывают на `T.nil`.[/дополнено]

#### Черная высота и гарантия балансировки

**Черная высота** $bh(x)$ — количество чёрных узлов на любом простом пути от узла $x$ до листа (не включая сам $x$).

**Ключевая лемма:** Красно-чёрное дерево с $n$ внутренними узлами имеет высоту:

$$h \leq 2 \log_2(n + 1)$$

**Вывод леммы (упрощённо):**

- Идеальное полное дерево высоты $h$ содержит $2^h - 1$ узлов, то есть $h = \log_2(n+1)$.
- В худшем случае красно-чёрного дерева пути чередуют красные и чёрные узлы: ЧКЧКЧК... Длина такого пути вдвое больше пути из одних чёрных узлов.
- Если черная высота равна $bh$, то минимальное число узлов в поддереве: $2^{bh} - 1$.
- Следовательно, $n \geq 2^{h/2} - 1$, откуда $h \leq 2\log_2(n+1)$.

**Практический вывод:** для $n = 10^6$ узлов $\log_2(10^6) \approx 20$, высота не превысит $40$ уровней — все операции выполнятся не более чем за $40$ шагов.

#### Операции поворота

Повороты — базовые операции для перестройки структуры дерева без нарушения свойства BST. Сложность: $O(1)$.

**Левый поворот вокруг узла $x$:**

```
    x                y
   / \    ===>      / \
  A   y            x   C
     / \          / \
    B   C        A   B
```

```python
def left_rotate(T, x):
    y = x.right            # y становится новым корнем поддерева
    x.right = y.left       # левое поддерево y переходит к x
    if y.left != T.nil:
        y.left.parent = x
    y.parent = x.parent    # обновляем родителя y
    if x.parent == T.nil:
        T.root = y
    elif x == x.parent.left:
        x.parent.left = y
    else:
        x.parent.right = y
    y.left = x             # x становится левым потомком y
    x.parent = y
```

**Правый поворот** — зеркальное отражение левого.

**Интуиция:** поворот — это локальная перестройка трёх узлов и трёх поддеревьев. Все свойства BST (левое < корень < правое) сохраняются, меняется лишь «кто является корнем» данного участка.

#### Вставка в красно-чёрное дерево

Вставка происходит в два этапа:
1. Вставить узел как в обычный BST, окрасив его в **красный**.
2. Восстановить инварианты с помощью перекраски и поворотов (`RB-INSERT-FIXUP`).

Три случая нарушения (когда родитель нового узла красный):

**Случай 1: Дядя красный**

Перекрашиваем родителя и дядю в чёрный, дедушку — в красный. Переходим к дедушке и продолжаем проверку.

```
        Дед(Ч)                   Дед(К)
       /      \      ===>        /      \
  Родитель(К)  Дядя(К)    Родитель(Ч)  Дядя(Ч)
       |                        |
      Z(К)                     Z(К)
```

**Случай 2: Дядя чёрный, Z образует «треугольник»**

Z является правым потомком родителя, а родитель — левым потомком дедушки (или зеркально). Делаем поворот вокруг родителя, переходим к случаю 3.

**Случай 3: Дядя чёрный, Z образует «прямую линию»**

Z является левым потомком родителя, родитель — левым потомком дедушки (или зеркально). Перекрашиваем родителя в чёрный, дедушку — в красный, делаем правый поворот вокруг дедушки.

```python
def rb_insert_fixup(T, z):
    while z.parent.color == RED:
        if z.parent == z.parent.parent.left:  # родитель — левый потомок
            y = z.parent.parent.right          # дядя
            if y.color == RED:                 # Случай 1
                z.parent.color = BLACK
                y.color = BLACK
                z.parent.parent.color = RED
                z = z.parent.parent
            else:
                if z == z.parent.right:        # Случай 2 -> переход к 3
                    z = z.parent
                    left_rotate(T, z)
                z.parent.color = BLACK         # Случай 3
                z.parent.parent.color = RED
                right_rotate(T, z.parent.parent)
        else:  # зеркальный случай (дядя слева)
            # ... аналогично, с заменой left <-> right
    T.root.color = BLACK
```

[дополнено]Среднее количество поворотов при вставке — около **2**. Перекраска может подниматься вверх по дереву, но это происходит только в случае 1; в случаях 2 и 3 алгоритм завершается немедленно после поворота. Поэтому общая сложность `RB-INSERT-FIXUP` — $O(\log n)$.[/дополнено]

**Типичные ошибки:**
- Забыть, что вставляемый узел всегда красный.
- Не обрабатывать зеркальные случаи (родитель — правый потомок дедушки).
- Не перекрашивать корень в чёрный в конце.

#### Удаление из красно-чёрного дерева

Удаление сложнее вставки: после стандартного BST-удаления нужно восстановить инварианты через `RB-DELETE-FIXUP`. Нарушение черной высоты возникает, когда удаляется чёрный узел.

[дополнено]Ключевая идея: при удалении чёрного узла $y$ дефицит черной высоты «навешивается» на замещающий узел $x$ как дополнительный «чёрный цвет». Четыре случая `RB-DELETE-FIXUP` последовательно избавляются от этого дополнительного цвета через перекраску и повороты.[/дополнено]

### 3.3 B-деревья (B-Trees)

#### Определение и механизм

B-дерево — это сбалансированное дерево поиска, обобщающее BST: каждый узел может хранить **несколько ключей** и иметь **несколько потомков**. Разработано для эффективной работы с дисковыми системами хранения.

Структура узла:

```c
struct BTreeNode {
    int n;             // количество ключей
    int keys[2*t-1];   // массив ключей (отсортированный)
    BTreeNode *c[2*t]; // указатели на дочерние узлы
    bool leaf;         // является ли узлом-листом
};
```

#### Параметр $t$ (минимальная степень)

$t \geq 2$ — минимальная степень B-дерева. Типичные значения в промышленных СУБД: $t \in [50, 2000]$ (один узел вмещает целую страницу диска).

Каждый узел (кроме корня) должен содержать:
- **минимум** $t - 1$ ключей (и $t$ указателей),
- **максимум** $2t - 1$ ключей (и $2t$ указателей).

Корень содержит от $1$ до $2t - 1$ ключей.

| Параметр | Формула | $t = 2$ | $t = 3$ |
|----------|---------|---------|---------|
| Мин. ключей в узле | $t - 1$ | 1 | 2 |
| Макс. ключей в узле | $2t - 1$ | 3 | 5 |
| Мин. детей у узла | $t$ | 2 | 3 |
| Макс. детей у узла | $2t$ | 4 | 6 |

[дополнено]При $t = 2$ B-дерево называют **2-3-4-деревом** (узел имеет от 2 до 4 потомков). Красно-чёрное дерево является изоморфным 2-3-4-дереву: каждый «чёрный кластер» (чёрный узел с его красными потомками) соответствует одному узлу 2-3-4-дерева.[/дополнено]

#### Пример B-дерева с $t = 3$

```
Корень:         [6, 12, 18, 24, 30]
                /    |    |    |    \
[1-5]   [7-11]  [13-17]  [19-23]  [25-29]  [31-35]
```

Все листья расположены на одной глубине $h$.

#### Теория и высота B-дерева

Высота B-дерева с $n$ ключами:

$$h \leq \log_t \frac{n + 1}{2}$$

[дополнено]Это значительно меньше, чем у красно-чёрного дерева при больших $t$. Например, при $t = 1000$ и $n = 10^9$ высота не превысит $3$. Именно поэтому B-деревья так эффективны: каждое обращение к диску читает целый узел, и при малой высоте количество дисковых операций минимально.[/дополнено]

#### Алгоритм поиска

```python
def b_tree_search(x, k):
    i = 0
    while i < x.n and k > x.keys[i]:
        i += 1
    if i < x.n and k == x.keys[i]:
        return (x, i)       # нашли
    elif x.leaf:
        return None         # ключ отсутствует
    else:
        # disk_read(x.c[i])  # загрузка с диска
        return b_tree_search(x.c[i], k)
```

Сложность: $O(t \cdot h) = O(t \cdot \log_t n)$ операций сравнения; $O(h)$ обращений к диску.

#### Алгоритм вставки

Вставка выполняется за один нисходящий проход. Ключевой принцип: **если узел переполнен (`2t - 1` ключей), его нужно разделить до спуска в него**.

```python
def INSERT(root, key):
    if root.full():            # корень полон — делаем новый корень
        new_root = split(root)
        insert_non_full(new_root, key)
    else:
        insert_non_full(root, key)

def insert_non_full(node, key):
    if node.is_leaf():
        # вставляем ключ в отсортированном порядке
        add_key_sorted(node, key)
    else:
        i = find_child_index(node, key)
        if node.c[i].full():       # дочерний узел полон
            split_child(node, i)   # разделяем его
            # после сплита средний ключ поднялся в node
            if key > node.keys[i]:
                i += 1
        insert_non_full(node.c[i], key)

def split_child(parent, idx):
    full_node = parent.c[idx]
    new_node = BTreeNode()
    mid = t - 1                        # индекс среднего ключа
    new_node.keys = full_node.keys[mid+1:]   # правая половина ключей
    full_node.keys = full_node.keys[:mid]    # левая половина ключей
    if not full_node.is_leaf():
        new_node.c = full_node.c[t:]   # правые потомки
        full_node.c = full_node.c[:t]
    # поднимаем средний ключ в родителя
    parent.insert_key(full_node.keys[mid])
    parent.insert_child(new_node, idx + 1)
```

Сложность: $O(h)$ дисковых операций, $O(t \cdot h)$ вычислительная.

#### Алгоритм удаления (три случая)

Удаление сложнее вставки: нужно гарантировать, что каждый узел на пути к удаляемому ключу имеет не менее $t$ ключей (чтобы после удаления соблюдался минимум $t-1$).

**Случай 1: Ключ $k$ находится в листе.** Просто удаляем $k$ из листа.

*Пример из слайда:* удаление F из листа — прямое удаление.

**Случай 2: Ключ $k$ находится во внутреннем узле.**

- 2a: Если левый потомок (предшественник $k$) имеет $\geq t$ ключей — заменяем $k$ его предшественником $k'$ и рекурсивно удаляем $k'$.
- 2b: Если правый потомок (последователь $k$) имеет $\geq t$ ключей — заменяем $k$ его последователем $k'$ и рекурсивно удаляем $k'$.
- 2c: Оба потомка имеют по $t-1$ ключей — объединяем их через $k$, рекурсивно удаляем $k$ из объединённого узла.

*Пример из слайда:* удаление M — предшественник M (ключ L) поднимается вверх и занимает место M.

**Случай 3: Ключ $k$ отсутствует во внутреннем узле $x$, нужный потомок имеет $t-1$ ключей.**

- 3a: Сосед потомка имеет $\geq t$ ключей — вращаем ключ из $x$ вниз в потомок, ключ из соседа — вверх в $x$.
- 3b: Оба соседа потомка имеют $t-1$ ключей — объединяем потомок с одним из соседей и ключом из $x$.

*Пример из слайда:* удаление G — G переносится вниз в узел DEGJK, затем удаляется из листа.

[дополнено]Ресурс для интерактивной визуализации вставки и удаления: https://www.cs.usfca.edu/~galles/visualization/BTree.html[/дополнено]

#### Применение B-деревьев в базах данных

Базы данных используют различные типы индексов в зависимости от задачи:

| Тип индекса | Применение | Поддержка диапазона |
|-------------|-----------|:---:|
| **B-tree (btree)** | По умолчанию; равенство и диапазон | Да |
| **Hash** | Только точное равенство | Нет |
| **GiST** | Геометрические и полнотекстовые данные | Зависит |
| **GIN** | Массивы, JSONB, полнотекстовый поиск | Частично |
| **SP-GiST** | Разреженные структуры (деревья, графы) | Зависит |
| **BRIN** | Очень большие таблицы с упорядоченными данными | Да |

**Ключевое преимущество B-деревьев:** один узел = одна страница диска. При чтении узла за одну дисковую операцию доступно сразу $2t - 1$ ключей и $2t$ указателей.

## 5. Справочный лист

### Красно-чёрное дерево — инварианты

1. Узел — красный или чёрный.
2. Корень — чёрный.
3. Листья (`NIL`) — чёрные.
4. Красный узел → оба потомка чёрные.
5. Все пути от узла до листьев содержат одинаковое число чёрных узлов.

### Красно-чёрное дерево — высота

$$h \leq 2\log_2(n + 1)$$

### Вставка в RBT — случаи fixup

| Случай | Условие | Действие |
|--------|---------|---------|
| 1 | Дядя красный | Перекрасить родителя, дядю в чёрный; деда — в красный; перейти к деду |
| 2 | Дядя чёрный, треугольник | Поворот вокруг родителя → переход к случаю 3 |
| 3 | Дядя чёрный, прямая линия | Перекрасить родителя и деда; поворот вокруг деда |

### B-дерево — ключевые формулы

| Параметр | Формула |
|----------|---------|
| Мин. ключей в узле | $t - 1$ |
| Макс. ключей в узле | $2t - 1$ |
| Мин. потомков | $t$ |
| Макс. потомков | $2t$ |
| Высота | $h \leq \log_t \dfrac{n+1}{2}$ |

### B-дерево — случаи удаления

| Случай | Ситуация | Действие |
|--------|---------|---------|
| 1 | Ключ в листе | Прямое удаление |
| 2a | Ключ во внутреннем узле, левый потомок «богатый» | Заменить на predecessor |
| 2b | Ключ во внутреннем узле, правый потомок «богатый» | Заменить на successor |
| 2c | Оба потомка «бедные» ($t-1$ ключей) | Слияние через удаляемый ключ |
| 3a | Ключ не в $x$, нужный потомок «бедный», сосед «богатый» | Вращение ключа |
| 3b | Ключ не в $x$, оба потомка «бедные» | Слияние узлов |

## 6. Практические советы

1. **Не пытайтесь запомнить все случаи поворотов наизусть.** Важнее понять, *почему* они нужны: нарушение инварианта → определить тип нарушения (треугольник или прямая) → применить нужный поворот. Навык рассуждения важнее механического запоминания.

2. **Тестируйте вставку/удаление на вырожденных случаях.** Отсортированная последовательность, обратно отсортированная последовательность, чередующиеся значения — BST ведёт себя совершенно по-разному. Всегда проверяйте крайние случаи.

3. **При реализации красно-чёрного дерева используйте сторожевой узел `NIL`.** Это избавляет от проверок `if (node != NULL)` в каждой операции и упрощает код; `NIL.color = BLACK` по определению.

4. **При работе с B-деревом помните про два прохода vs. один проход.** Вставка с проактивным разделением (один нисходящий проход) эффективнее, чем вставка с последующей коррекцией снизу вверх, потому что исключает повторное чтение узлов с диска.

5. **Выбирайте структуру под задачу:** если данные в памяти — красно-чёрное дерево (используется в `std::map`, `std::set` в C++, `TreeMap` в Java). Если данные на диске или индекс базы данных — B-дерево (PostgreSQL, MySQL InnoDB). Если нужен только поиск по равенству — хэш-таблица быстрее.
